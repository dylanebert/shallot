# Render Contract

Exports surface in `.claude/rules/exports.md`.

## Layering

Peer modules, each with one job:

- **`render/`** — universal contract. `Frame` + `View` + `Lighting` + `Camera` + `Render` singleton + frame loop, plus the three registries `Surfaces` (`{ name, bindings?, preamble?, interpolators?, vs?, fs? }` — shading only, mesh-agnostic), `Meshes` (`{ name, vertices, indices, indexBase, indexCount }`), and `Draws` (`{ name, surface, mesh, args }`). `RenderPlugin` lives here, owns the `Lighting` uniform + the `AmbientLight` / `DirectionalLight` / `PointLight` (+ the `Spot` cone add-on, + the `Volumetric` light-shaft marker the fog reads) components; `BeginFrameSystem` packs the Lighting UBO each frame (`writeLighting` from the singleton-entity components). Render also owns the **froxel cluster substrate** (`cluster.ts`): a 16×9×24 log-Z grid of per-view view-space cluster AABBs, built by one compute pass (`ClusterSystem`) only when a view's projection changes (`BeginFrameSystem` packs the per-view params; the staging-vs-last compare is the dirty signal) and published as `"clusterAabbs"` — the grid the light cull bins into and later froxel consumers (fog, decals) read. **Point lights are GPU-driven end to end** (also `cluster.ts` — grid + binning are one substrate module; the deliberate deviation from Bevy's CPU light assignment): `PointLight` fields are slabs, and the per-frame `LightCullSystem` pass pair runs a **compact** (membership-gated scan over capacity → the compacted world-space light list, position from the transforms firehose, hex color decoded + intensity baked GPU-side; the compacted entry's third `params` lane carries the sphere-source `radius` (its sign the `Volumetric` opt-in flag — a negated radius scatters in the fog march; the lit path reads `radius²`, sign-immune) plus the spot cone for a `Spot` light — the cone axis is the entity's forward oct-packed, the angular `scale`/`offset` Frostbite's `getAngleAtt` from the inner/outer half-angles, `(radius, 0, 0, 1)` for a plain point so the FS angular factor is 1; cap `MAX_POINT_LIGHTS` with a non-silent warn) then a **cull** (one thread per cluster per view, lights LDS-batched + view-transformed once per workgroup, sphere-vs-AABB, atomic-append into a flat index pool with `lightGrid` recording each cluster's offset+count; pool overflow counts into a header word surfaced by a throttled 8-byte readback — never silent). `BeginFrameSystem` stages each slot's world→view matrix (`LightCull.viewStaging`) for the cull's view transform. The pure TS oracles (`clusterAabb` / `zSlice` / `sliceDepth` / `lightClusters`, in `render/core`) are what the gym Mirror asserts pin the WGSL twins to (the `render` scenario's `clusters` + `light-cull` + `light-overflow` checks). Render **defines** the registries but **never iterates** them — they're bridges between any producer and any consumer renderer. The `Meshes` contract is a slice descriptor: a mesh points into the quantized vertex streams (the 16 B `vertices` main + the 8 B `position` depth stream + the per-mesh `quant` table, gpu.md rule 6) + index GPU storage, with an `indexBase` offset and `indexCount` length. Static meshes register through `mesh()`, which stages the f32 typed arrays and — at `RenderPlugin.warm`, via `flushMeshes()` — `quantizeMeshes` packs them all into **one shared family buffer set**, each mesh a slice (its own `indexBase` + a meshId keying its `quant` entry, indices shifted by its vertex base). So sear binds geometry once across all static meshes, and the layout is `multi-draw-indirect` ready. Procedural producers (terrain) own their `GPUBuffer`s and register slices directly. Meshes sharing a buffer set share a bind group in sear.
- **`part/`** — one dogfooded producer. `PartPlugin` + `Part` + `Color` components, the built-in `cube`/`sphere`/`capsule` meshes via `mesh()` (the static-mesh helper — stages typed arrays, packed into the shared family buffer at `RenderPlugin.warm`), pack compute pass, `Parts` GPU publication. Part is **renderer-independent**: it publishes per-instance data (`transforms`, `color`, `eids`) but registers **no surface**, so it carries no lighting model and renders under any consumer. `Part.surface` defaults to the *name* `"default"` and `Part.mesh` to `"cube"`; sear ships the `default`/`unlit`/`vertex` materials, so the bare happy path renders (see "Surface authoring"). The pack is a **per-view frustum cull → count → scan → scatter**: it groups by `(surface, mesh)` pairs indexed `mid * surfaceCount + sid`, and runs the spine once per active camera (`gid.y` = view slot), gating count + scatter on the instance's world bound vs that view's `frustums[slot]` (the per-mesh local sphere from `Mesh.bounds`, center transformed by the instance transform, radius by the largest |scale| axis). A slot `>= viewCount` (headless, or the synthetic slot 0 when no camera exists) keeps everything — culling degrades to plain compaction. Output is **slot-major** (`slot * pairCount + pair`): each camera owns its own `drawArgs` records + `capacity`-sized `packedEids` region, so sear draws each view's survivors via `Draw.args.viewStride`. **Robust to runtime mesh registration — meshes need NOT be known at warm.** Surfaces are WGSL shading programs declared in code (never data-driven), so `Surfaces.size` is final at warm and the surface count is the one axis baked into the pack shaders. Mesh + view counts arrive late (terrain grids, cameras attaching), so both come from a per-frame `cullParams` uniform — the pipelines compile **once** at `PartPlugin.warm` and never recompile. `drawArgs` + `counts` + `packedEids` + `meshBounds` size lazily and **grow** when a mesh registers or a camera attaches (`syncBuffers`, called each frame — two int compares, not per-entity tracking): `drawArgs`/`counts` scale with views × pairs, `packedEids` with views × capacity, `meshBounds` with mesh count. Because the pair index is `mid * surfaceCount + sid`, a new mesh only **appends** pair slots, so existing offsets hold; `syncBuffers` reallocates the affected buffers, re-runs `registerDraws` (re-seeds every (slot, pair) static arg + repoints the Draws at the grown buffer), and frees the old buffers behind the submit fence. `registerDraws` registers one Draw per `(instanced-surface × mesh)` pair — *every* combination, no entity query — and seeds each slot's static args (`indexCount: mesh.indexCount`, `firstIndex: mesh.indexBase`, `baseVertex: 0`). A surface is instanced when it declares the `eids` + `transforms` bindings. The per-frame pack fills `instanceCount` + the compacted `firstInstance`; pairs no entity uses pack to `instanceCount: 0` and `drawIndexedIndirect` no-ops them. So a producer registers a mesh + surface (in `initialize` or later — terrain registers its grid-sized mesh in setup) and spawns Parts — the draw it needs already exists, no producer-side draw code, whenever its entities appear. (Pre-MDI this is `O(surfaces × meshes)` draws, the unused ones cheap no-ops; `multi-draw-indirect` folds the per-pair fan-out onto the GPU.) `PartPlugin` depends on `RenderPlugin` + `SlabPlugin`.
- **`sear/`** — the one consumer, one renderer, one plugin (`SearPlugin`): a niagara-style raster forward renderer, one color pass (opaque draws then `blend` draws composited over them, fused into a single `beginRenderPass`; see "Camera passes") + an opt-in single-sample prepass (per-camera lanes: `Tag` → id, `Depth` → depth), with **sun shadows sampled inline in the FS** — a sear-internal feature data-gated on the `Shadow` component on a directional light, not a separate layer or plugin (placement + light-camera lifecycle in the sear-internal `sear/shadows.ts`; mechanism in "Sun shadows" below). Owns vertex pull: declares `vertices` in its own WGSL preamble and pulls attributes from it, while binding the mesh's index buffer as a **hardware index buffer** (`setIndexBuffer`) and issuing `drawIndexedIndirect` — so `@builtin(vertex_index)` is the resolved vertex index and the post-transform vertex cache dedups (the VS runs once per vertex, not once per index). The mesh's `indexBase` flows through as `firstIndex`. Compiles a pipeline per Surface uniformly — no "Part-shaped" detection. Sear's VS preamble declares `eid`, `world`, `worldNormal` plus each declared interpolator as mutable locals (identity / zero defaults), splices the **standard instance transform** for any surface declaring `eids` + `transforms` bindings (the instancing convention — sets `eid = eids[iid]` and applies `transforms[eid]`), then the surface's own `vs` chunk, then standardizes the output. **Lighting is explicit, not forced**: sear exposes `lit(baseColor, normal)` + `lightFactor(normal)` (diffuse) and `litPbr(Pbr, normal, world)` (metallic-roughness Cook-Torrance) helpers (reading the `Lighting` uniform); the fs chunk writes the final color `col` and shades by calling them (or not — unlit just writes the base color). Iterates Draws, dispatching each at indirect offset `Draw.args.offset + view.slot * Draw.args.viewStride` — so a per-view-culled producer draws each camera's own records, while a view-independent draw (no `viewStride`) is unaffected. Depends on `RenderPlugin` only. **Never imports from `part/`** — verify by grep.
- **`glaze/`** — the default postfx composite + the postfx chain. `GlazeSystem` runs one **compute dispatch** per camera reading `view.framebuffer` (the offscreen a renderer drew into) and `textureStore`ing `view.present` (the swapchain, a base-format storage view), so a renderer never writes the swapchain directly — that's what gives postfx a rendered color to read. Compute, not a render pass, so present costs no tile load/store on TBDR (`gpu.md` "Render passes on TBDR") — the same path a consumer's fused composite takes. The swapchain isn't sRGB (storage views can't be), so glaze encodes linear→sRGB via the shared `LINEAR_TO_SRGB_WGSL`. Renderer-agnostic: it queries every camera, imports only `render/core`, so sear (MSAA-resolved) and a custom renderer (single-sample) composite the same way. A renderer orders itself ahead with `before: [GlazeSystem]`; **glaze never imports a renderer** (the direction is render ← glaze, never glaze → sear). The chain (a scene-referred grade → `Tonemap` operator → post-saturation → OkLab-L posterize/dither → γ-2.2 vignette) is per-camera config via the `Glaze` component, packed **slot-major** into a per-view uniform. The tonemap is the analytic operator set in `standard/glaze/tonemap.ts` (Khronos PBR **Neutral** the default, plus ACES Fitted, AgX, Reinhard, Reinhard-luminance, SomewhatBoring, None — `Tonemap` enum); **index 0 = Neutral**, so a camera with no `Glaze` still tonemaps Neutral, the zero-config default display transform. The tonemap rolls off true highlights because sear renders into an **HDR `rg11b10ufloat` offscreen** (single path, no flag — see "Camera passes") rather than clamping the brights at store. No post-process AA — MSAA in sear is the geometric AA baseline. Glaze is the default composite: `SearPlugin` forces none, so a consumer registers `GlazePlugin` (Neutral tonemap + present) or ships its own. A consumer needing a **scene effect before the tonemap** adds a compute system in the post-color seam that transforms the scene via `sceneTransform` (`render/core`): it reads the current `view.framebuffer`, writes a scratch, and repoints `view.framebuffer` so the next effect — or glaze — reads its output. `sceneTransform` hands back a **ping-pong pair**, so two effects chain (the returned `read`/`write` are always the distinct halves of a per-camera scratch pair; `b` allocates lazily, and a third in-frame consumer is the trigger to grow a real ring). The seam has two kinds, ordered by the **`OverlaySystem` anchor** (`render/core`, a no-op system `RenderPlugin` registers): **scene-space transforms** (fog) run `before: [OverlaySystem]`, **screen-space overlays** (outline) run `after: [OverlaySystem]` — so an overlay composites on top of the transformed scene regardless of plugin registration order (`Color → fog → overlay → outline → glaze`), the two effects referencing the anchor instead of each other. An overlay composites through `sceneTransform` **as a compute pass too** (like glaze), never a render pass into `view.framebuffer`: the framebuffer may be a storage-only `rgba16float` scratch (no `RENDER_ATTACHMENT`) or the `rg11b10ufloat` offscreen, so a render-pass overlay would assume a format/usage it can't guarantee. No registry — plain system ordering + the anchor, like a producer's compute.

A camera selects its renderer by the marker it carries (`Sear`). A consumer can author its own renderer against `render/` — a custom pipeline (e.g. a magenta-clear pass) selected per camera by its own marker, since nothing engine-side is special-cased to sear. Other producers (compute-driven terrain, particle systems, custom UI overlays) are peers of `part/` — they plug in by calling `Meshes.register(...)`, `Surfaces.register(...)`, and `Draws.register(...)` against `render/`. Procedural meshes skip the `mesh()` helper — they create their own `GPUBuffer`s (compute-written each frame, or pre-built) and register slice descriptors into them. The renderer never asks where the buffers came from. No `part/` involvement; sear iterates uniformly.

## Registration

The producer/extension API (in `render/core`, not the happy-path barrel) is the `Surfaces` / `Draws` / `Meshes` registries (instances of `Registry<T>` from `@dylanebert/shallot`). Producers call `.register(spec)` (returns a stable numeric ID) and `Draws.delete(name)` directly — no separate wrapper functions. `mesh(spec)` is the one exception: it allocates GPU buffers from typed arrays before calling `Meshes.register(...)` for you.

The `Surfaces` / `Draws` / `Meshes` maps live in `render/core` for renderer/producer extension authors and internal iteration. There is no `part/core`: the per-instance transform is a **convention** sear applies (a surface declaring `eids` + `transforms`), not a Part export, and `Parts` (the pack's GPU output) is internal to Part. The public barrel (`@dylanebert/shallot`) is the pure-data happy path — `mesh()`, the components (`Part`, `Color`, `Camera`), and the plugins (`RenderPlugin`, `PartPlugin`, `SearPlugin`). Sun shadows are a sear-internal feature gated on the `Shadow` component on a directional light (exported beside the `Tag` camera marker); RT shadows return on native WebGPU RT. Surface authoring is the developer API, reached through the `*/core` subpaths.

## Surface authoring

`Surfaces.register({ name, bindings, preamble, interpolators, vs, fs })`. A surface is shading only — it names no mesh. Geometry is chosen per-Draw (`Draw.mesh`); for Part entities that's the per-entity `Part.mesh` field, so one surface shades any mesh and the pack groups draws by `(surface, mesh)`. `vs` and `fs` are **WGSL chunks** the consumer renderer splices into its own `@vertex` / `@fragment` bodies. `preamble` is an optional WGSL chunk spliced at module scope after the binding declarations — use it for helper functions, structs, or constants the entry chunks call, since WGSL forbids function declarations inside function bodies. Surfaces that share helpers across sites export them as `WGSL` string constants and template-concat into `preamble`. **Sear already splices `OCT_ENCODE_WGSL` (`octEncodeNormal` / `octDecodeNormal`), `POS_QUANT_WGSL` (`MeshQuant`, `decodePos` / `decodeUv` / `meshIdOf`), and `XFORM_WGSL` (`Xform`, `xformPoint` / `xformNormal` / `xformMat` — the decomposed transform-firehose decode) at module scope for every surface** — the oct chunk for the per-vertex *storage* normal decode (the interstage + VAT normals are both plain vec3, never oct — gpu.md rule 9), the quantized vertex decode, and the instance-transform reconstruct — so a preamble must not redefine any of them (a duplicate definition fails to compile). `interpolators` (`Record<name, WGSL type>`) are custom vertex→fragment varyings: sear adds them to its output struct after the built-ins (cap 16 total), exposes them as mutable locals the `vs` chunk writes, and rebinds them as locals the `fs` chunk reads — integer types get `@interpolate(flat)` automatically. The **built-in interstage set is per-surface, not a fixed 0..4**: `eid` + `world` + the normal (interpolated as a plain `vec3` `worldNormal`, renormalized in the fs — not oct across the interstage; gpu.md rule 9) always cross; `uv` + `localPos` cross **only when the `fs` chunk reads them** (an unread varying is pure per-fragment bandwidth — gpu.md rule 9), so custom interpolators pack from whatever the built-in count is. The fs still sees `worldNormal` / `world` / `eid` (and `uv` / `localPos` when it uses them) as plain `vec3` / `vec3` / `u32` locals — the renormalize is sear's, invisible to the chunk. This is the path for per-vertex lighting (compute `lightFactor` in `vs`, multiply per fragment), tangent frames, and the like. Sear's preamble declares `eid: u32`, `world: vec4<f32>`, `worldNormal: vec3<f32>` (and each interpolator) as mutable locals before splicing `vs`; the chunk can override any of them. A surface declaring `eids` + `transforms` storage bindings gets sear's standard per-instance transform automatically (the instancing convention) — no `vs` chunk needed for the transform. The renderer owns: Frame/View/Lighting uniforms, binding declarations, the quantized vertex decode (it dequantizes the stream into `localPos` / `localNormal` / `uv` before the `vs` chunk — gpu.md rule 6) + the `VertexOut` struct, the vertex pull, the instance transform, the `lit` / `lightFactor` lighting helpers, and the entry points.

Sear's prelude in scope when authoring chunks:

- `vertices: array<vec4<u32>>` — the quantized vertex-attribute pull binding (the 16 B main stream: unorm16 pos + meshId / oct normal / unorm16 uv, gpu.md rule 6), sear-declared. A `vs` chunk almost never reads it directly — sear decodes it into the `localPos` / `localNormal` / `uv` locals first (against the meshId-selected `meshQuant` table). The depth/shadow pipelines bind the 8 B `position` stream at the same slot instead. The buffer comes from `Meshes.get(draw.mesh)`. The mesh's index buffer is bound by sear as a **hardware index buffer** (`setIndexBuffer` + `drawIndexedIndirect`), so `@builtin(vertex_index)` is the resolved vertex index — there's no surface-visible `indices` binding
- `eids: array<u32>`, `transforms: array<Xform>`, `color: array<u32>` — per-instance bindings a producer publishes by name (Part does). `transforms` is the decomposed firehose (`{pos, quat, scale}`); reconstruct with `xformPoint` / `xformMat` (`XFORM_WGSL`, spliced for every surface). A surface declaring `eids` + `transforms` is **instanced** — sear applies the standard per-instance transform (the convention below); `color` is a material attribute the default surfaces read (an sRGB-packed LDR u32, the `srgb8x4` slab — decode with `unpackLdrColor`, which sear splices for every surface)
- `frame: Frame`, `view: View`, `lighting: Lighting` — universal uniforms. `view` carries `viewProj`, `resolution`, and the normalized camera basis `right`/`up` (billboard surfaces orient from it; forward = `-cross(right, up)`; in the shadow pass it's the light camera's, so billboards face the light). `lighting` carries `ambientColor` (`rgb` + intensity in `a`), `sunDirection` (normalized), `sunColor` (intensity pre-baked into rgb); `RenderPlugin` packs from the singleton `AmbientLight` / `DirectionalLight` components each frame
- `vidx: u32`, `iid: u32` — vertex + instance indices
- `eid: u32`, `world: vec4<f32>`, `worldNormal: vec3<f32>`, plus each declared interpolator — mutable locals in vs scope (identity / zero defaults). For an instanced surface (declaring `eids` + `transforms`) sear sets them from the standard transform (`eid = eids[iid]`, `world = xformPoint(transforms[eid], world.xyz)`, normal via `xformNormal` — the inverse-transpose, correct under non-uniform scale); the surface's own vs chunk runs after and can override
- `clipPos: vec4<f32>` — present only for a `screen: true` surface: its vs chunk writes the clip position itself and sear emits `out.clip = clipPos` (a world-space surface leaves it, and sear projects `view.viewProj * world` *after* the chunk, so displacing `world` still projects). `view.resolution` (pixels) sizes constant-pixel screen geometry; sear disables back-face culling for `screen` surfaces (their quads have no consistent winding). The lines producer is the worked example
- `localPos: vec3<f32>`, `localNormal: vec3<f32>`, `uv: vec2<f32>` — vertex attributes, mutable, may be modified before the chunk writes `world`
- In fs scope: `eid`, `uv`, `localPos`, `worldNormal`, `world` (world-space position, `vec3`), and each interpolator are rebound as locals; `fin` is the `VertexOut`. Write the final color into `col: vec4<f32>` (defaults to opaque black); sear returns it verbatim. Shade explicitly via `lit(baseColor, worldNormal)`, `lightFactor(worldNormal)`, or the metallic-roughness `litPbr(Pbr(albedo, metallic, roughness, occlusion, dielectric), worldNormal, world)` (all read the `lighting` uniform). **Unlit** = write the base color into `col` and never call a lighting helper. **Per-vertex** = compute `lightFactor`/`litPbr` into a custom interpolator in `vs`, read it per fragment. **Per-pixel** = call `lit`/`litPbr` in `fs`. The mode is which chunks the surface ships — sear neither forces nor selects it
- `litPbr(s: Pbr, normal, world) -> vec3` — the metallic-roughness Cook-Torrance model (GGX + Smith + Schlick) over a **half-Lambert diffuse lobe** (the soft default below), the shading model sear's `default`/`vertex` materials + the glTF importer use. `Pbr` is `{ albedo, metallic, roughness, occlusion, dielectric }`; `dielectric` is the non-metal base reflectance (F0) — `0` (the engine `Material` default) gives **zero specular** for non-metals (the flat shallot look, reducing `litPbr` to `lit` exactly at metallic 0 / roughness 1), `0.04` (glTF's choice) gives the spec-standard dielectric Fresnel. Only the diffuse is wrapped; the specular keeps the physical clamped cosine, so metals + glTF dielectrics stay correct. View-dependent, so it reads the camera world position from `view.eye`. Shares the sun-shadow + clustered-point seam (`sunVisibility` / `pointScale` privates) with `lightFactor`, so a `vs`-side call gets neither (same as the diffuse path).
- `lit(baseColor, normal) -> vec3`, `lightFactor(normal) -> vec3` — module-scope helpers sear provides. `lightFactor` is `ambient + sun·halfLambert·shadow + point-light sum` (callable in `vs`); `lit` applies it to a base color (`baseColor * lightFactor(normal)`, per-fragment). The diffuse cosine is **Valve half-Lambert** (wrapped + squared): it fills the terminator instead of a hard plastic falloff, applies to every diffuse term (sun + point), and is deliberately non-energy-conserving — the stylized happy-path default. The point-light term is **clustered**: `clusterOf()` maps the fragment (`fragCoord.xy` tile + view-depth log slice, from `view.cluster` = near/far/perspective/slot) to its froxel, and the loop reads only that cluster's lights through sear's `lightGrid` + `lightIndices` bindings (group-0 slots 6–7, GPU-written by the render-side light cull) with Bevy's smooth-windowed inverse-square falloff — the TS oracle is `distanceAttenuation` in `render/lighting.ts`; its seam is `fragWorld`/`fragCoord`/`pointScale` privates the color-FS scaffold fills, so per-vertex shading and the prepass entries get a zero point contribution, same shape as the shadow seam. The shadow seam is `sunVisibility`, a `var<private>` defaulting to 1.0 that multiplies the **sun term only** (ambient stays unshadowed); the fragment scaffold fills it by calling `sampleSunShadow(world, worldNormal)` — projecting the fragment's world position into the group-1 shadow map and PCF-comparing — before splicing the chunk. So `lit()` in `fs` receives sun shadows + point lights for free; **per-vertex shading** (`lightFactor` in the `vs`, before `world` is known) gets **neither** — it reads the defaults (sun fully lit, zero point contribution). No casting light (no `Shadow` on the sun) → sear binds a 1×1 fallback with `enabled: 0` → `sampleSunShadow` returns 1.0 → fully lit, identical to shadows-off

`Part.surface` defaults to the *name* `"default"`, and **sear ships it** — Part is renderer-independent (it registers no surface), but the renderer owns its default materials. `SearPlugin.initialize` registers `"default"` (per-pixel metallic-roughness via `litPbr` + emissive), `"vertex"` (per-vertex Gouraud — `litPbr` in the `vs` into a `litColor` interpolator), and `"unlit"` (`col = vec4(unpackLdrColor(color[eid]).rgb, 1.0)`). `default`/`vertex` declare `eids` + `transforms` + `color` + the **`material`** binding — sear's `Material` component (`{ params: slab(f16x4, "material") }`, GPU mirror `vec4<f16>`, lanes metallic/roughness/emissive/occlusion, authored named via `laneAlias` e.g. `material="metallic: 1; roughness: 0.2"`). f16 keeps emissive an unbounded HDR glow strength; its flat default (metallic 0, roughness 1, dielectric 0) is **zero specular**, so a bare Part shades exactly like the pre-PBR diffuse `lit`; `unlit` omits `material`. No `part/` import: the transform is the convention, `color` + `material` resolve by name. External authors do the same — `Surfaces.register({ name, bindings: { eids, transforms, color }, fs: ... })` — then pick geometry per entity in the scene: `<a part="surface: x; mesh: y" />`, where `y` is any registered {@link Mesh}.

Part registers one Draw per `(instanced-surface × mesh)` pair — every combination, no entity query — re-running as meshes register (`syncMeshes`). An instanced surface declares the `eids` + `transforms` bindings. Pairs no entity uses pack to zero instances and `drawIndexedIndirect` no-ops them; non-instanced surfaces (registered + drawn by other producers) coexist on equal footing.

## Draw shape and the compute → drawing handoff

```ts
interface Draw {
    name: string;
    surface: string;
    mesh: string;
    args: { indirect: GPUBuffer; offset?: number; viewStride?: number };
}
```

`surface` selects shading; `mesh` selects geometry (looked up in `Meshes` at bind-group build time); `args` is **always indirect** — `indirect` is a GPU buffer holding WebGPU's 20-byte `DrawIndexedIndirect` record (`{indexCount, instanceCount, firstIndex, baseVertex, firstInstance}`) at `offset`, and the renderer binds the mesh's index buffer (`setIndexBuffer`) and issues `pass.drawIndexedIndirect(indirect, offset + view.slot * viewStride)`. `viewStride` (default 0) makes the record per-view: a producer that culls into a separate record per camera lays them out slot-major, and the renderer indexes the active camera's record. Omit it for a view-independent draw. Everything is GPU-driven, so a CPU-known draw `writeBuffer`s its record into a small buffer rather than passing literals (no `draw()` direct variant). For static meshes, `indexCount` is the mesh's index count and `firstIndex` its `indexBase`; `baseVertex` is 0 (indices are absolute vertex positions), so `@builtin(vertex_index)` is the resolved vertex index and sear pulls `vertices[vidx]` with hardware vertex reuse. Producer's compute writes the indirect buffer; in-encoder ordering guarantees the subsequent draw sees those writes.

`Draw` carries no resources. Geometry resolves through `Meshes`; surface bindings resolve through `Compute.buffers` (see below).

**System ordering.** Producer compute systems and renderer systems both declare `after: [BeginFrameSystem]` and live in the "normal" subset of the draw group. A producer whose per-frame compute writes buffers the renderer reads for **position** — vertex/index geometry, or the `transforms`/`eids` of an instanced surface — must *also* declare `before: [PrepassSystem]`. `PrepassSystem` is the **geometry-emit ordering anchor** (and the opt-in prepass): it runs first among the geometry passes, so it's the single point separating "geometry written" from "geometry read." Sear reads that geometry across several passes per frame — the prepass, the shadow map (light view), and the color pass all read it within the frame — so an emit landing *between* them desyncs the reads (a static re-emit survives, so the failure surfaces the moment the geometry animates or an edit lands). `before: [PrepassSystem]` pins the emit ahead of all of them in one edge; `after: [BeginFrameSystem]` alone doesn't, and registration order is no substitute. Buffers read only in the fragment stage (color, material params) stay `after: [BeginFrameSystem]`. `EndFrameSystem` is the **sole** `last: true` system — it submits the encoder. Adding `last: true` to a renderer ties for sort order with EndFrame and silently breaks rendering (EndFrame submits before the renderer records anything).

**`Draws.register` timing.** Calling `Draws.register(...)` from a `warm()` hook is unsafe — plugin warms run in `Promise.all` (parallel, not dep-order), so slab `.gpu` buffers and producer-registered named buffers may not yet exist in `Compute.buffers`. Capture in `System.setup` (first-frame) instead — by then every warm has completed. (Part's own internal `registerDraws` runs later still, from `syncMeshes` in `PartSystem.update`, so it's safe to re-run as meshes grow.)

## Binding resolution via `Compute.buffers`

Surfaces declare bindings by name + WGSL type. The actual GPU buffer is resolved at bind-group build time against `Compute.buffers: Map<string, GPUBuffer>` (engine-level, on the `Compute` singleton). Two paths populate it:

- **Named slabs auto-publish.** `slab(f32, "pulse")` registers `.gpu` under name `"pulse"` once `Slab.prepare` runs (during `SlabPlugin.warm`). Unnamed slabs stay invisible to renderers.
- **Producers publish their own.** Part registers `Parts.packedEids` as `"eids"`; `TransformsPlugin` publishes its transform firehose as `"transforms"`. The color slab is named `slab(srgb8x4, "color")` so it auto-publishes.

Texture + sampler bindings resolve from sibling registries by binding type, same lazy pattern: `texture-2d` / `texture-2d-array` / `texture-depth-2d` from `Compute.textures: Map<string, GPUTexture>`, `sampler` / `sampler-comparison` from `Compute.samplers: Map<string, GPUSampler>`. Producers publish a loaded texture (`Compute.textures.set("brick", tex)`) or sampler the same way they publish buffers; the surface declares `brick: { type: "texture-2d" }` + `samp: { type: "sampler" }` and sear binds a default texture view at bind-group build time (a `texture-2d-array` binding gets a `2d-array` view; the glTF importer's albedo array is the worked example). Until the named texture exists the draw is skipped (bind group resolves to null), so an async texture load in `warm` is fine.

Geometry is the exception — `vertices` + `indices` resolve through `Meshes` (selected by `Draw.mesh`), not `Compute.buffers`. They are sear-declared, not surface-declared.

A surface authoring a binding `pulse: { type: "storage", element: "f32" }` works the moment its `"pulse"`-named slab is a *registered component's field* — only those are collected and published, so a standalone `slab(f32, "pulse")` made in a plugin's `initialize` never allocates. No `inputs` plumbing, no producer awareness. Sear's bind-group cache invalidates on resource-identity change (buffer, texture, sampler, or the resolved Mesh).

## Per-Draw CPU cost is the WebGPU floor

Sear does `setPipeline + setBindGroup + setIndexBuffer + drawIndexedIndirect` per Draw per camera — once in the color pass, and (when the camera carries a lane marker) once in the prepass (see "Camera passes" below). Both passes share one bind group; every pipeline compiles from one shader module per surface. Bind groups cached on buffer-identity, pipelines cached per surface. CPU cost scales `O(#(surface, mesh) pairs in use)`, not `O(#instances)`.

`multi-draw-indirect` (D3D12 `ExecuteIndirect`, Vulkan `vkCmdDrawIndirectCount`) is not in WebGPU spec ([gpuweb#1354](https://github.com/gpuweb/gpuweb/issues/1354)). When it ships, sear consolidates internally; the contract doesn't change.

**One Draw per `(surface, mesh)` pair, not per entity.** The Part producer registers one Draw per `(Part-surface × mesh)` pair — the pack folds the N Parts sharing a pair into one `instanceCount`, sear dispatches once. Unused pairs pack to zero and no-op. A user with one surface + the cube sees one live drawIndexedIndirect for the whole scene. Producers that register one Draw per entity would blow up the CPU loop — the architecture doesn't push you that way.

## Camera passes

Sear renders each view with a 4× MSAA (default — per-camera `Camera.antialias`, below) **color pass** (`depthCompare: greater` + write — reverse-Z, near→1/far→0, depth clears to 0 — its **own** depth) that resolves into the offscreen, plus — only when the camera carries a lane marker (`Tag` / `Depth`) — a 1× **prepass** that owns its own single-sample depth (cleared to 0 + `greater` + write) and emits the camera's requested lanes. There is **no default-path depth round-trip**: with sun shadows sampled inline (the forward FS projects each fragment into the shadow map itself), the bare camera (no lane marker) runs no prepass at all, and the color pass clears + writes its own depth directly (TBDR-friendly). `PrepassSystem` is also the geometry-emit ordering anchor (see "System ordering") — it runs first among the geometry passes, so even a bare camera leaves it in the schedule for producers to pin `before:`.

Every surface compiles **two shader modules** sharing the same generated vertex math (`surfaceCode(surface, variant)`): the **color** module (the color fs + the real group-1 shadow bindings) and the **prepass** module (the prepass entry points with shadow **stubs**, no group-1 declarations) — so the color, prepass, and shadow passes still run identical vertex math. The split is forced, not stylistic: a surface chunk's `lit()` statically reaches `pointShadowOf` (the clustered loop's per-light shadow factor), so a single module would pull group 1 into the prepass pipelines (which bind group 0 alone) and make the shadow/atlas passes sample the very depth they're writing. The prepass and color passes each own their own depth (`greater` + write) and never cross-compare, so no `@invariant` is needed on the clip builtin.

**MSAA is the geometric AA — per-camera and runtime-toggleable, sample-based only** (never post-process blur: no FXAA / SMAA, no TAA). `Camera.antialias` (default on) selects each camera's mode: on → the color pass's opaque + transparent draws render at `SAMPLE_COUNT` (4) and resolve (hardware `resolveTarget`) into the single-sample offscreen once at pass end; off → they render single-sample straight into the offscreen, no resolve (the crisp / pixel-art path, pairs with `build({ pixelRatio })` below 1). It's a scene attribute or a live `Camera.antialias.set(eid, 0)`; sear compiles the color/transparent pipelines per sample count (`colorPipelines` — the single-sample twin lazily on first no-AA draw via `ensureSingle`) and allocates the matching targets (`colorTargets`). The prepass, shadow map, and screen-space effects stay single-sample regardless (an id can't MSAA-resolve, a render pass can't mix counts), so they own their own 1× depth. Coverage-correct (silhouettes only, texel interiors crisp). The scene offscreen is **`rg11b10ufloat` (HDR), single path, no flag** — every camera, no variant or runtime branch — so glaze's tonemap rolls off radiance >1 instead of clamping it; the MSAA color target shares the format. `rg11b10` (4B) **halves** the MSAA color-target + resolve bandwidth vs `rgba16float` (8B) — the dominant `sear:color` cost at 4× MSAA, measured ~0.2ms → ~0.1ms (flat-grey Sponza, 1440p, 4090) — for ~3% relative precision (no alpha; over-blending needs no dst alpha). It needs the `rg11b10ufloat-renderable` feature for multisample + resolve, on the base floor (`CLAUDE.md` — the whole target set has it; `rgb10a2uint` can't substitute, integer formats are neither blendable nor resolvable). The one storage-written HDR buffer, the `sceneTransform` scratch, stays `rgba16float` (`rg11b10` isn't storage-capable). `rgba16float` is the higher-precision HDR format; if a scene bands in smooth gradients it returns as a per-app format option (trigger-gated, not built).

**Backdrop seam — a fullscreen background draw fused into the color pass.** `Backgrounds.register({ name, bindings?, preamble?, fs })` (`sear/core`, the `Surfaces` analogue) is a *view-ray → HDR color* recipe; the per-camera `Backdrop` component selects one by name. Sear draws it as a **fullscreen triangle inside the MSAA color pass, after the opaque draws and before the blend draws**, at the reverse-Z far plane (clip z = 0) with `depthCompare: greater-equal` + **no depth write** — so an un-rendered pixel (cleared depth 0) passes `0 >= 0` and a geometry pixel (depth > 0) fails: the backdrop fills only background pixels, opaque geometry overdraws it, and blended draws composite over it, **with no depth readback or ping-pong**. Under MSAA the per-sample depth test resolves the sky↔geometry silhouette antialiased for free, and it lands in the HDR offscreen before fog + tonemap. The fs reconstructs the world-space view ray `dir` per-pixel from `@builtin(position)` + `view.invViewProj` — **not** an interstage interpolator (the ray is derivable from the pixel, so crossing it would waste a varying — gpu.md rule 9; the `fog` march reconstructs world the same way). No `Backdrop` on a camera → the flat `Camera.clearColor` is the only backdrop (the opt-in fallback). The engine names no sky concept — a plugin (`extras/sky`) owns its own sky math behind this seam. This is decisively better than the `sceneTransform` compute seam for an *opaque* backdrop; fog uses that seam because it integrates across the shaded scene, sky does not.

**Opt-in prepass lanes — a closed engine-owned union, not a consumer registry.** The prepass emits per-camera lanes gated by markers (Bevy's `DepthPrepass` / `NormalPrepass` shape): each lane is data — `{ name, marker, format, clear, fs-local }` — that `surfaceCode` + the prepass iterate. Two ship: the **depth** lane (the depth-stencil, marker `Depth`) and the **id** lane (a color attachment, marker `Tag`). A camera carrying any marker runs **one** prepass regardless of lane count — the marker set selects the attachment list + the pipeline variant (compiled per lane-set, bounded since the union is engine-closed), not the pass count. A consumer reads each lane as a declarative per-camera handle (`Views.get(eid).depth` / `.tag`, sibling of `view.framebuffer`, `null` until the prepass has run that lane).

**One shape per camera output, classified by sampling semantics — not an output grab-bag.** Three classes:

- **color** — `view.framebuffer`, the per-camera **offscreen** scene-color target (sampleable, `Render.format`), the only blendable, postfx-fed output. Sear's 4× MSAA color pass (opaque + transparent draws) resolves into it once; glaze composites it to `view.present` (the swapchain). The shaded lane (opaque draws then `blend` draws) is uniformly **color-only** (`@location(0)`): opaque / `clip` / `alpha` differ only in blend state + depth policy, never in fragment output shape. No MRT.
- **id** — `view.tag`, single-sample, opaque-only, discrete (no average of an identity is meaningful). A prepass color lane (`PrepassSystem`, in the `after: [PrepassSystem], before: [ColorSystem]` seam — the prepass *is* the anchor), opt-in per camera via sear's `Tag` marker, filled over the opaque + `clip` set (`r32uint`, `TAG_NONE` clear, the prepass's own depth `greater` + write); `alpha` is excluded (a transparent pixel has no single owner). The u32 id can't ride the color pass: WebGPU can't multisample an integer format and a render pass can't mix sample counts, so a u32 id can't share the 4× MSAA color target — the separate single-sample prepass is forced, not a choice (Bevy hits the same wall, force-disabling MSAA for its packed-id deferred path; sear keeps forward+MSAA and isolates the id to the prepass). It's Unreal's `CustomStencil` generalized to a u32 lane (Bevy's prepass carries no id — it CPU-raycasts). Never blended, never multisampled, never a color-pass MRT attachment.
- **depth** — `view.depth`, the prepass's single-sample depth, stored + published only when the camera carries `Depth` (a `Tag`-only camera tests depth but discards it — TBDR keeps it in tile memory); the within-frame input for a screen-space effect (AO / fog / outline). `null` for a camera that doesn't request it.

The tag is **surface-authored**: a mutable fs local symmetric with `col`, defaulting to the entity's `eid` for an instanced surface and `TAG_NONE` otherwise, fs-overridable (a custom producer authors its own — terrain → `capacity + cell`, reconstructed from `localPos`: the id lane runs in the position-only prepass, so `uv` is 0 and `worldNormal` the +Z default there — author the tag from `world` / `localPos` / `eid` only). It's **opt-in per camera** via sear's `Tag` marker (in the spirit of Bevy's `DepthPrepass` / `NormalPrepass`): omit it and the id lane is absent, allocating no target; a consumer that reads `view.tag` (hover, outline, picking) adds `Tag` to its camera. A consumer reads `view.tag` for hover readback / outline / debug, the same declarative per-camera handle as `view.depth`; the engine owns the u32 + the `TAG_NONE` sentinel, the meaning is a consumer convention, and the engine knows nothing of picking. **The rule that stops the drift:** sear core owns color; the prepass owns the depth + id lanes; every other screen-space camera output is a **lane** in that one prepass filling a named `view.*` field. A future g-buffer output (normals, motion) is a **new lane** — a `COLOR_LANES` table row + a `view.*` field + a `surfaceCode` conditional, emitted as one more MRT attachment in the **same** prepass, never a new pass. Don't accrete `view.*` fields speculatively, don't grow the *color* pass into MRT, and don't flatten the lanes into a generic `Map<string, texture>` that would obscure the singular-depth / single-id / single-color distinction.

## Transparency: `blend` modes

A surface declares its render mode in one `render/core` `Surface` field, `blend` (alongside `bindings` / `interpolators`): omitted = opaque, `"alpha"` = translucent, `"clip"` = masked-opaque cutout. The trio maps to Unity's Opaque / Transparent / Cutout. `render/` and `part/` only carry the field — mode is per-surface (every instance shares it), not a Part component, so an entity changes mode by pointing `Part.surface` at a surface with that mode. The cull / pack spine is unchanged: every instance culls + packs the same way; only the routing differs. Sear compiles one pipeline shape per mode:

- **opaque** (default) — 4× MSAA single-target color pass (its own 4× depth, `greater` + write) + the 1× prepass id lane (`view.tag`, the prepass's own depth, when the camera carries the `Tag` marker).
- **`"alpha"`** — one non-opaque pipeline, drawn in the color pass after the opaque draws (`ColorSystem`, the single fused geometry pass): 4× MSAA, the same MSAA color + depth the opaque draws wrote, resolved into the offscreen once at pass end, a single color target (`Render.format`), `depthCompare: greater-equal` against the opaque depth, **depth-write off**, straight-alpha blend (gamma-correct — the sRGB blend unit linearizes; the fs writes linear `col` with alpha). No id lane — a transparent pixel has no single owner, so a blended surface is never a hover / outline target, and it has no prepass pipeline at all. Occlusion is the depth-*test* only: nearer opaque geometry hides it, it occludes nothing, and with no prepass depth pipeline it doesn't cast. A clipped *decal* (the ring stamp) is this mode + `discard` + alpha 1 — depth-tested, casts nothing.
- **`"clip"`** — opaque (writes depth, single-target color, stamps `view.tag` under the `Tag` marker), but its empty-lane-set prepass fragment (`fsPrepass`, no color target) runs the chunk so a `discard` cuts holes the shadow map (which renders through that same pipeline) honors — a clipped solid casts a holed shadow. The foliage / fence case.

**The GPU sort is deferred, not skipped**: single-layer / non-overlapping `"alpha"` transparency is exact unsorted (a convex mesh's back-face cull leaves one layer per pixel). The back-to-front sort lands with the first consumer that has *overlapping* `"alpha"` instances, keyed on camera-space depth.

## Culling lives in the producer, not the consumer

Culling is folded into the producer's compaction (Part's pack), not added as a sear pass. The pack already reads each instance's transform + pair and writes the compacted list; the cull test is one predicate inside that same pass. Putting it in sear would mean a *second* compaction (read the packed list, re-test, re-pack) and would force sear to learn each producer's instance→bound mapping, costing it the agnosticism that makes per-Draw CPU cost `O(pairs)`. Sear stays blind to instance semantics; uncull is correct-but-slower, never a correctness bug. The per-view cull volume is a **tagged descriptor** in the shared `Render.cullVolumes` buffer (a header `vec4` with the tag in `.x` + the 6 frustum planes), packed each frame by `BeginFrameSystem` from each view's viewProj and tested in the pack's hot `visible()` (tag `CULL_FRUSTUM` = a 6-plane AND). Every view frustum-culls — cameras, the sun, and each point/spot shadow combo (its own frustum-culled depth view, `frustumVolume`). It rides one storage binding (the pack's scatter sits at the 10-storage ceiling — gpu.md). **When a second many-instance producer wants culling**, extract the `cull → count → scan → scatter` spine into a shared `render/core` helper (that second consumer reveals the right shape) — do NOT relocate culling into sear.

**Occlusion culling (Hi-Z) is deliberately absent — domain-driven, not universal.** Hardware early-Z already rejects occluded *fragments* for free; Hi-Z only adds rejecting occluded *geometry processing*, which pays off only at high occluded depth complexity behind high-poly meshes (dense interiors / cities — artist-authored density). Idiomatic shallot scenes (procedural, open, scatter, terrain, viz) have low occlusion, so a depth pyramid + per-instance reprojection test rejects ~nothing while costing every frame, worse on the integrated floor — and it would force retaining + storing depth on the default path, regressing the no-depth-round-trip ("Camera passes"). The procedural answer is gen-time visibility (don't emit structurally-occluded geometry — surface-only meshing already does this; authored cell/portal streaming), not a universal runtime pyramid. Revisit *only* gated on a real shallot-target scene that exhibits high occluded depth complexity, and prefer authored streaming over engine Hi-Z even then.

## Shadows: sear-internal, gated on the `Shadow` component

Sun shadows are a sear-internal feature, gated on the `Shadow` component on a directional light — like the `Tag` marker on a camera, presence is the switch. Add `Shadow` to the sun and the builtin surfaces cast + receive with no producer-side code; omit it and sear renders fully lit (no atlas allocated). Sear renders the cascade atlas through its own depth pipelines and samples it inline in the FS — no `Visibility` seam, no separate plugin (Bevy's clustered-forward shape: light-data-gated, sampled in the FS). `Shadow.distance` is the CSM **max shadow distance** (the far bound of the last cascade). The CPU/ECS half (cascade-camera placement + lifecycle) is `sear/shadows.ts`; the GPU half (atlas, render, group-1 binding) is `sear/index.ts`.

**The sun joins the unified culled-combo spine — N cascades, not a special case.** For a **perspective** main camera the directional shadow box is split into N cascades (default 4, `SunShadows.cascades`) along the camera's view range out to `Shadow.distance` — the practical/PSSM split (`cascadeSplits`, three.js/Bevy λ≈0.5), each a frustum-slice bounding-sphere-fit ortho view (`cascadeFit`, texel-snapped per cascade). An **orthographic** main camera has uniform texel density, so depth cascades buy nothing: it gets a **single** box fit to the visible ground footprint (`orthoFootprintFit`, the three.js/standard ortho special-case — the frustum-slice fit doesn't reach an ortho camera's visible ground), `cascadeComboEids` length 1, a sentinel far-bound so the receiver always selects it. Either way every box extends its near plane **toward the light** by `Shadow.distance` (`placeFromCenter`'s margin — Bevy pushes the directional near plane to ∞, three.js's finite `lightMargin`), so a tight near cascade still captures occluders above its slice instead of clipping them (the multi-cascade boundary-bleed fix). Each box is a pooled off-screen depth-only ortho `Camera` (`attachView`, the point combo-pool shape — `cascadeComboEids`), created lazily in play mode the first casting frame, taking one of the `CASCADE_RESERVE` cull slots. So `BeginFrameSystem` packs each cascade's viewProj + frustum like any camera and **the Part pack culls casters into each cascade independently** — sun + point + spot share one culled-combo path. `ShadowCameraSystem` (simulation) poses them via `updateCascades` each frame from the sun + the main camera. (Gated by the gym `render` `cascade` / `cascade-ortho` / `cascade-boundary` probes.)

**Per-frame sequence**: `ShadowCameraSystem` (simulation; no-op when no `Shadow`) poses the cascade cameras → `BeginFrameSystem` packs them → producers pack → `PrepassSystem` (the ordering anchor; resolves the shared draw list) → `ShadowMapSystem` (`after: [PrepassSystem], before: [ColorSystem]`) renders into a **dedicated cascade atlas** (a separate `depth32float` texture from the point atlas — Bevy's directional/point split), each cascade a tile (`tileTransform` fold) through sear's compiled empty-lane-set prepass pipeline. Like the point atlas, a two-pass **re-gather** (`sear:cascaderegather`, the shared `sear/regather.ts` helper instantiated for both atlases) concatenates each casting mesh's per-cascade culled members into **one `drawIndexedIndirect` per casting mesh**, then writes the per-cascade `SunShadow` uniform (each cascade's viewProj + atlas rect + far-bound + texel world size + the bias knobs). Sear's forward FS selects a cascade by **linear view-z** (Bevy `get_cascade_index`, from `view.eye` + the camera basis), **3×3 PCF**-samples that cascade's atlas tile (taps clamped to the tile interior), and blends into the next cascade across the `SunShadows.overlap` band (`sampleSunShadow`). No `Shadow` → no atlas, sear binds its `enabled: 0` fallback = fully lit. Acne: a `normalBias` per-cascade-texel normal offset + a residual `depthBias` lift, both `Shadow` knobs.

**Point-light shadows: the same gate, an importance-sized depth atlas.** `Shadow` on a `PointLight` opts that light into shadows — same presence switch, same sear-internal shape (`distance`/`resolution` don't apply; coverage is the light's `range`). A `Shadow` on a **`Spot`** rides the same atlas as a **single-tile** perspective caster (the cone FOV, one tile) instead of six cube faces — so a spot costs one atlas tile, not six — and `pointShadowOf` branches on the caster's stored cone tangent (0 = point), reconstructing the spot receiver from the lookAt basis (`right`/`up`/`fwd`) with the same analytic perspective formula. Storage is **one fixed square depth atlas sub-allocated by importance** (PlayCanvas's `LightTextureAtlas` — the same `texture_depth_2d` + comparison-sampler binding shape the sun uses, no cube-array support needed, fits the integrated floor): each caster claims power-of-two square tiles — six for a point, one for a spot — sized so tile **area tracks** its apparent contribution (`intensity · range² / dist²`; side ∝ √score, so the hero light's tiles are large and distant lights' small), buddy/quadtree-packed largest-first (no fragmentation for power-of-two squares). The atlas + the `pointCasters()` cap are the `PointShadows` config — `{ atlas, casters, hysteresis }`, default 2048 × 8, `atlas` + `casters` fixed before `build()` like `capacity` (ceiling `MAX_POINT_CASTERS`). Over the cap the **highest-importance** lights win, never query order, with a hysteresis margin (`PointShadows.hysteresis`, live-tunable) keeping a casting light its slot unless a challenger beats it by that margin (no flicker as the camera moves); a caster that won't fit even the smallest uniform tiling is dropped with a non-silent warn (`packCasters` retries after dropping the least important). The 90° face frustum is **widened by the PCF margin** (the PlayCanvas seam fix — `pointTanHalf(tilePx)` per tile size, so the seam margin is a constant number of texels at every tile, and the receiver divides by the same per-tile tangent so a face-boundary direction lands inside the tile), near/far = `range/1000` / `range`. The atlas renders in **one pass, one `drawIndexedIndirect` per casting mesh**. Each combo (a point's cube face, a spot's cone) is its own **frustum-culled depth view** — a pooled off-screen camera (`pointComboEids`) the Part pack culls casters into through the same `cull → count → scan → scatter` spine every view uses, so a member rasterizes only the faces it actually hits (no over-amplification on the integrated floor). A two-pass **re-gather** compute (`sear:pointregather`) concatenates each casting mesh's per-combo *culled* members into one contiguous mesh-major run + a per-instance combo index — packed `(combo << shift) | eid` at the surface's `eids` lane, so the point pipeline gains **no** storage binding past the 10-ceiling (gpu.md) — and writes one `DrawIndexedIndirect` record per casting mesh (Pass A sums the per-combo counts + prefixes the bases, Pass B scatters, one compute pass; Dawn injects ~1µs of untimed validation per indirect draw — `gpu.md`). The point VS reads each instance's `(eid, combo)` from that packed list at the `eids` lane, looks up the combo's `(caster, face)` in `comboMeta`, projects by that combo's CPU-computed **tile viewProj** — the allocated atlas-UV rect folded into clip space (`tileTransform`), so the hardware does the perspective divide + near-plane clip and the depth matches `pointShadowOf`'s analytic receiver (a manual `fc.xy/fc.w` in the VS couldn't clip a triangle behind a face's near plane) — and discards fragments outside that face's `pointTileRects` rect (indexed `caster·6 + face`, the seam discard). **Only an instanced surface casts** — the re-gathered list keys on per-instance `eids` + `transforms`. `updatePointShadows` (sear/shadows.ts) ranks the casters + sizes/packs the tiles (`packCasters` over a buddy `createPacker`), computes the combo tile viewProjs CPU-side (`tileTransform × perspective × lookAt`, dense per active combo — a point's six from the `POINT_FACES` table that also generates the `pointFaceOf` WGSL twin, a spot's one from its cone) + the `comboMeta` + the per-(caster, face) rects, and poses one pooled depth camera per combo for the cull. The combo views share the `MAX_SLOTS` pool with the shading cameras: `MAX_COMBO_SLOTS = MAX_SLOTS − MAX_VIEWS − CASCADE_RESERVE` caps them, dropping the least-important caster with a loud warn if exceeded. `ShadowMapSystem` writes the `PointCaster` uniform (light pos + source eid, near/far + bias, spot basis; published `"pointShadows"`) + the `pointTileRects` uniform (the allocated atlas-UV rects; published `"pointTileRects"` — both pinned by the gym `render` `point-shadow` check). The FS hook is **inside the clustered loop**: the compact pass bakes each light's source eid into the compacted entry's `color.a`, and `pointShadowOf` matches it against the caster slots (an empty slot's pos.w = −1 matches nothing), picks the cube face from the light→fragment direction, reads that face's tile rect, reproduces the face projection's depth analytically (pinned to `perspective()` by unit test, so the hardware depth compares exactly), and 3×3 PCF-compares in the tile — **clamping each tap to the tile interior** so a grazing or wide-spot sample never bleeds into a neighbour tile (the leak fix; shadow sampling runs only for the fragment's cluster shortlist). Receiver bias mirrors the sun's *effect* — a normal offset scaled by the distance-growing face texel plus the `depthBias` residual — but applies the residual in **linear** depth (`pointReceiver` biases the view distance before the perspective remap); a fixed subtraction in the sun's hyperbolic NDC depth would grow with z² and peter-pan under perspective. No `Shadow` on any point light → no atlas allocated, no combo cameras spawned. **Off-screen views don't bin lights, by slot order**: `BeginFrameSystem` packs shading views (presenting cameras) into the low slots `[0, MAX_VIEWS)` and depth-only views above them (up to `MAX_SLOTS`); the cluster + light-cull passes dispatch over `Render.shadeCount` alone, so a depth-only slot never allocates cluster state or bins into the shared index pool (binning them would overflow it and drop random clusters' lights, visible as froxel flicker).

**Regression set.** The shadow pipeline is gated by the gym `render` scenario, one mode per run — `bun bench --scenario render --param mode=<m>`: `cull` (the `cull → count → scatter` spine — `per-combo-cull` / `per-cascade-cull` counts vs the CPU frustum oracle, the over-draw cut, one-draw-per-mesh, the `point-shadow` tile no-leak), the framebuffer-probe shaded rows `spot` / `spotShadow` / `pointShadow` / `cascade` / `cascade-ortho` / `cascade-boundary` / `acne`, and `zfight` (reverse-Z precision). Run the modes a change touches. There's no run-all sweep — a separate Playwright session per mode is slow and flake-prone (vite 500 / navigation races under WSL), so the targeted per-mode bench is the gate.

## Producer pattern

For a custom producer (terrain, particles, anything compute-driven):

1. `Surfaces.register({ name, bindings, preamble?, vs?, fs? })` — shading only, no mesh (geometry is per-Draw). The declared bindings are what the producer's per-instance buffers satisfy by name; `preamble` is an optional module-scope WGSL chunk for surface helpers.
2. Allocate own `GPUBuffer`s for the **three vertex streams** (the 16 B `vec4<u32>` main + the 8 B `vec2<u32>` position + a per-mesh `MeshQuant` table), indices, indirect args, per-instance data. A compute writer emits the quantized format with `POS_QUANT_PACK_WGSL`'s `encodePos` / `encodeUv` + `octEncodeNormal`, against a `MeshQuant` of the mesh's analytic AABB — splice `POS_QUANT_WGSL` + `POS_QUANT_PACK_WGSL` + `OCT_ENCODE_WGSL` (the voxel mesher + kitchen ground are the worked examples). Register slices via `Meshes.register({ name, vertices, position, quant, indices, indexBase, indexCount })` — same contract `mesh()` uses for the static case. Sharing buffer refs across many registrations with different `indexBase` offsets is how a producer expresses "all these meshes live in one atlas" — sear groups draws using the same buffer set. Indices are absolute vertex positions; producers writing into a shared atlas pre-shift their indices at write time. Publish any per-instance buffers under a name in `Compute.buffers` (or use named slabs that auto-publish).
3. `Draws.register({ name, surface, mesh, args: { indirect: argsBuf } })`. Geometry resolves from `Meshes`; the mesh's `indexBase` flows through as `firstIndex` in indexed-indirect args (its index buffer needs `GPUBufferUsage.INDEX`); surface bindings resolve from `Compute.buffers`; sear caches the bind group keyed on the resolved buffer identities.
4. Add ECS systems with `after: [BeginFrameSystem]` that mutate buffer contents each frame via compute passes encoded on `Render.encoder`. Buffer **contents** change per frame; **identities** don't. A system writing position-determining geometry (vertices/indices, or an instanced surface's `transforms`/`eids`) must *also* declare `before: [PrepassSystem]` so every sear geometry pass (prepass, shadow map, color) reads the same geometry (see "System ordering").

No imports from `part/` or `sear/` needed — just `render/core` for the registries. Bind-group cache invalidates only on buffer-identity change (rare — destroy + recreate, not contents update).

## Don't

- Don't import from `sear/` in `render/` or `part/`. Direction is render ← (part / sear / custom producers), never the reverse. Sear never imports from `part/` either — they're peers, both consume from `render/`.
- Don't import a renderer (`sear/`, custom) from `glaze/`. Glaze imports only `render/core` and composites whatever a renderer left in `view.framebuffer`; a renderer imports `GlazeSystem` to declare `before: [GlazeSystem]` (renderer → glaze for ordering is fine; glaze → renderer is not). Don't write the swapchain (`view.present`) from a renderer — render into the offscreen (`view.framebuffer`) and let a composite (glaze, or a consumer's own) present it, or postfx has nothing to read.
- Don't register one Draw per entity. One Draw per *surface*; pack the entities into its `instanceCount`.
- Don't use `setVertexBuffer` — vertex *attributes* are a storage binding (vertex pull). Indices ARE a hardware index buffer: sear binds `Meshes.get(draw.mesh).indices` via `setIndexBuffer` and issues `drawIndexedIndirect`, so the post-transform vertex cache dedups (the VS runs once per vertex, not once per index). The attribute pull and the hardware index buffer are orthogonal — a producer's index buffer needs `GPUBufferUsage.INDEX`, and its indirect record is the 20-byte `DrawIndexedIndirect` shape.
- Don't recreate buffers per-frame for stable resources — buffer identity invalidates sear's bind-group cache. Allocate once, mutate contents.
- Don't put renderer-specific code in `render/` or `part/`. Lighting model, pipeline composition, the shadow-map render — sear's concern.
- Don't reintroduce a shadow seam. Shadows (sun + point) are sear-internal, gated on the `Shadow` component — sear owns the light cameras, the map/atlas renders, the `SunShadow` / `PointCaster` params, and the group-1 bindings (`sampleSunShadow` / `pointShadowOf` + the `sunVisibility` default + the `enabled: 0` / empty-slot fallbacks). No `Visibility` singleton, no separate shadow plugin; the map is light-data-gated like Bevy's. A *new* screen-space effect that feeds the color pass (AO, fog) is still an external plugin in the `after: [PrepassSystem], before: [ColorSystem]` seam (one compositing *over* the resolved color, like an outline, slots `after: [ColorSystem]` instead) — sun shadows fold in because the technique is settled (shadowmap, sampled inline), not an invitation to fold every effect into sear.
- Don't mark a renderer system `last: true`. EndFrameSystem owns that slot; sharing it ties in sort order and EndFrame wins on registration order.
- Don't call `Draws.register(...)` from `warm()` — slab `.gpu` may not be allocated yet (parallel warms). Use `System.setup`.
