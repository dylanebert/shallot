# Physics substrate

`standard/physics` is the backend-neutral substrate every physics backend plugs into: the shared authoring components (`Body` / `Spring` / `Joint` + their traits, `ShapeKind`, the `Hulls` registry), the CPU `raycast` + `pick` layer, the kinematic-character controller (`standard/character`, its `Player` composition in `standard/player`), and a thin typed backend handle (`PhysicsBackend`) a backend plugin installs at `Physics.backend`. It owns **no simulation** â€” a backend does. Two backends ship: `standard/avbd` (the massive-scale GPU AVBD solver â€” the mechanism rule is `avbd.md`) and `standard/tumble` (the default: the built-in wasm rigid-body engine â€” `tumble.md`, glue and engine layer both). A scene swaps engines by flipping one manifest plugin line.

## The substrate decision â€” a typed core plus escape hatches, not a union interface

The substrate is **shared components + a thin typed backend handle**, not conventions alone and not a union interface. This is the load-bearing call, proven when the second backend landed:

- **Not conventions-only.** Before the substrate, "physics" *was* AVBD â€” `PhysicsStep` and its GPU buffers directly. A second backend with the same component surface but a wasm world has no shared type to install against; every consumer (the character sweep, `pick`, the compose) would branch on which engine is live. The typed `PhysicsBackend` handle is the one seam they all read (`Physics.backend.readBody` / `.setKinematic` / `.compose`), so a consumer never knows which backend answers.
- **Not a union interface.** A union of everything both backends can do is dishonest â€” sensors, mesh colliders, tumble's 9 joint types have no AVBD counterpart; AVBD's GPU-scale pile has no tumble analogue. An *intersection* (only what both share) straitjackets both. So the handle is the **atomic core** every backend must serve â€” pose read, kinematic write, velocity write, step, compose â€” and **everything richer rides that backend's own singleton, used imperatively**: `Avbd.step` (GPU pipelines, the grab), `Tumble.world` (the live wasm `World`, all 9 joint configs). Escape hatches, not interface growth. A new capability that only one backend has is a method on *its* singleton, never a new `PhysicsBackend` member.

This rule is the design record's residue (the working spec was deleted at ship).

## The backend contract

- **`Body` / `Spring` / `Joint` are the same objects across backends** (idempotent registration, `ecs.md` "Stable component ids"), so their traits (`bodyTraits` / `springTraits` / `jointTraits`) live in `standard/physics` once, not duplicated per backend plugin. `Body.excludes [Transform]` (single-writer: `Body` carries the spawn pose + mass properties on slab, the backend owns the GPU/world pose after spawn; `mass â‰¤ 0` = static). Authoring detail is "Authoring" below.
- **`PhysicsBackend` is a plain-object handle** (`standard/physics/index.ts`) a backend plugin installs in its `warm()` via `installBackend(handle)` and removes in `dispose()` via `uninstallBackend()`. `installBackend` **throws if one is already installed** â€” a scene runs exactly one physics backend at a time (the single-backend guard; two backend plugins in one manifest is the loud failure). `Physics.backend` is `null` until a backend warms; a consumer that runs before warm guards on it.
- **The substrate owns the schedule contract, the backend owns the mechanism.** `StepSystem` / `ConstraintSystem` / `ComposeSystem` (this file) are the ordering anchors; each delegates to the installed handle (`Physics.backend?.step()`, etc.). A producer that must run before/after the solve orders against `StepSystem`, not against a specific backend. `ConstraintSystem` derives the backend-neutral `SpringDef` / `JointDef` lists from a scene's `Spring` / `Joint` entities and hands them to `PhysicsBackend.setSprings` / `.setJoints`, re-uploading only when the authored set's signature changes (`installBackend` re-arms this on a fresh backend, so a `warm()` that installs the handle can't forget the two-step). Richer per-backend joint authoring (motors, a world anchor, tumble's cone/twist) rides that backend's escape hatch, not `JointDef`.
- **`readBody(eid)` may be up to one fixed tick stale.** A GPU backend serves a `Mirror` snapshot (a frame-old pose is fine for its consumers); a CPU backend serves this tick's fresh state. Consumers (the character sweep, `pick`) treat the staleness as contractual.

## The two-writer firehose partition

A `Body`+`Part` renders at the **physics-owned** pose: the backend's `ComposeSystem` (`after: [BeginFrameSystem]`, `before: [PrepassSystem]`) scatters the live pose into the `transforms` firehose for the renderer to read, no `Transform` component. This puts **two writers on the `transforms` buffer** â€” the `Transform` compose (render, `standard/transforms`) and the physics compose â€” and the contract between them is **slot ownership, never write order**:

- Each firehose slot is written by exactly one composer. The Transform compose is **membership-gated** to the entities that carry `Transform`; `Body` excludes `Transform`, so the two writers partition the slot space cleanly (a `Body`'s slot is physics-owned, a `Transform` entity's slot is render-owned).
- **Why ownership, not order:** a CPU backend uploads its pose via `queue.writeBuffer`, which lands in queue order **before** the frame command encoder runs. So a Transform compose that scattered *all* capacity slots (not just its members) would stomp every physics-owned slot after the backend's writeBuffer already wrote it â†’ every physics body renders at the stale/zero Transform pose (the stage-7b tumble bug: zero-scale invisible bodies; latent since a CPU backend first shipped â€” AVBD was immune because its compose is an encoder pass ordered *after* writeBuffer, masking the missing gate). An encoder-ordered backend hides the bug; a writeBuffer-ordered one exposes it. Gate both by ownership: pinned by the gym `ragdoll` floor-pixels check + a headless stub-device compose test (`standard/tumble/tumble.test.ts`).

## Kinematic character

The character is a `mass â‰¤ 0` capsule `Body` whose pose a **CONTROLLER owns, not the solver** â€” one shared substrate primitive, never a per-backend fork. The f64 spec is `tests/avbd/character.ts` (the executable spec the runtime CPU sweep reproduces, exactly as the AVBD oracle is the spec for its solver); no physics reference covers it â€” it's the standard kinematic-controller recipe (Unity/Godot/Jolt) grounded on the rounded/hull closest-point narrowphase (`avbd.md`).

**The runtime controller is the CPU sweep (`standard/character`) â€” there is no GPU character pass.** `CharacterSweepSystem` runs `sweepCharacter` (`sweep.ts` â€” the f32-tier twin of the oracle, validated against it by `character-sweep.oracle.ts`) for every `[Character, Body]` each fixed tick in the FIXED group, `before: [StepSystem]`, so it lands before the backend solve. The result is the inputâ†’poseâ†’camera latency the pillar targets: a same-frame CPU path with no GPU readback. `PlayerSnapshotSystem` (`standard/player`, fixed, `after: [CharacterSweepSystem]`) snapshots `pose` off the CPU `CharState` so it captures THIS tick's swept pose, then renders `lerp(prev, curr, fixedAlpha)` â€” the only camera lag is the kept one-tick interpolation + the display fence.

The coupling to the backend is **one-way, through `Physics.backend`** â€” the sweep never forks per engine:

- The sweep writes the character's fresh pose as a kinematic body: `Physics.backend.setKinematic(eid, pos, quat, false, realizedVel)` (the moving-platform mechanism; the explicit `realizedVel` excludes the cosmetic ground snap so the carry-of-riders + broadphase pad read the swept motion). Backend dynamics then collide against the CURRENT-tick player.
- The sweep reads the static collision world from the authored `Body` slab, and dynamic / moving-platform candidates one fixed tick stale off `Physics.backend.readBody` (a frame-old dynamic pose is fine).
- The push is the **full-CPU apply**: a shoved dynamic's new velocity goes straight to the backend via `Physics.backend.setVelocity` (which also wakes it), no GPU character work. A thin GPU-additive apply was considered and left unbuilt â€” the difference is only the perpendicular component of the pushed body's velocity staleness (â‰ˆ 0 for a clean push); don't re-propose it without a measured pile-push artifact.

**How each backend honors this** is that backend's rule: the AVBD-solver features that keep an incidental kinematic contact inert (the all-static dual-ramp gate, the character owning no broadphase blocks) are `avbd.md`; tumble handles a kinematic body natively.

**Two recorded boundaries** (the substrate character's honest limits): a backend's escape-hatch-only geometry (a collider created directly on `Tumble.world`, never mirrored to a substrate `Body`) is **invisible to the sweep** â€” the sweep reads the substrate `Body` slab + `readBody`, not the backend's private world; and there is **no swept TOI** against such geometry.

**Validation** is the deterministic CPU oracle (no GPU char pass to single-step against): `character-sweep.oracle.ts` (f32 sweep == f64 oracle â€” slide / step-up / coyote / jump / slope / push / carry / cull-equivalence) + `character.oracle.ts` (the oracle's own feel + conservation tests). The backend **coupling** (setKinematic uploads the pose, the backend pushes the box / carries riders) is the real-GPU gym `character` live scene; the inputâ†’camera latency is the gym `character` `probe=1` gate.

The controller algorithm, shared by the CPU sweep (`sweep.ts`) and the f64 oracle (`character.ts`):

- **Collide-and-slide (move-then-depenetrate); the push-out normal is the geometric CLOSEST-POINT MTV, not a SAT reference-face normal.** The MTV is the closest point between the capsule CORE segment and the polytope (`closestPointOnHull` â†” closest-on-segment alternation). At a step edge the closest-point normal tilts UP, so the rounded bottom climbs a sub-radius step for free; a SAT face normal would wedge the capsule horizontally. This is why the character builds on the rounded/hull narrowphase yet uses the closest-point primitive, not the manifold SAT.
- **Three subtleties that are decisions, not tuning:** (1) gravity integrates ONLY while airborne (gated on last tick's `grounded`) â€” applying it every tick leaves a tangential residual after the perpendicular push-out, so the capsule creeps even on flat-enough ground; gating it holds a walkable slope and lets a too-steep one slide. Gravity is **per-character** (`Character.gravity`, default 0 = the world gravity the sweep reads via `Physics.backend.gravity`) â€” a player falls/jumps snappier (e.g. `âˆ’50`) than rigid bodies without touching the world's `âˆ’10`. (2) ground-snap pulls a grounded capsule onto the surface (gap 0, glued on stairs/slopes) but is SKIPPED while rising (`realized.y > 0`), or it pulls the capsule back off a step it's climbing. (3) `grounded` keys on a walkable contact within a snap band, so a settled capsule reads grounded at gap 0 without flickering. The realized velocity (`(sweptPos âˆ’ start)/dt`) feeds the backend broadphase pad + the carry; the snap is excluded from it.
- **The sweep is culled â€” the oracle `gather` verbatim, on the CPU.** `gather` keeps only bodies whose bounding sphere can reach the capsule within this tick's motion budget (a derived contact-set-preserving sphere superset: every phase gates on `gap < GROUND_SNAP` or `depth > 0`, so a culled body is output-neutral), so cost is independent of world body count. Candidates keep scan order â€” the order-dependent selections (first-max depth, last-walkable groundNormal) are bit-identical to a full scan. Loud guards (`SweepDiag`): `overflow` (the per-character cap `MAX_CHAR_CANDIDATES` = 64, first 64 in scan order) + `guard` (travel exceeded the band's budget â€” the spawn-inside class). The `cull: false` brute seam is bit-identical (the cull-equivalence tests pin culled == brute EXACT, f64). A cached-plane (Jolt `SolveConstraints`-style) slide was REJECTED: the free step-up depends on RE-PROBING at the penetrated pose â€” a start-pose cached plane pushes the char straight back, and Jolt only climbs stairs via a separate `WalkStairs` layered on top; don't re-propose plane caching without it.
- **Jump = coyote + buffer, single jump.** A jump fires only when BOTH the coyote credit (refilled to `COYOTE_TIME` while grounded, decays airborne) and the jump buffer (set by `jump()`, decays) are positive, and CONSUMES both â€” a held/spammed button can't re-fire mid-air. Pinned by oracle tests + the gym `character` jumper station.
- **Full-speed push = velocity transfer.** Each tick, for every dynamic the char is in **contact** with (gap within the `GROUND_SNAP` touch band â€” the sweep leaves the char AT the face, so penetration is never the trigger), the push drives that body's velocity along the push normal up to the char's **desired** into-speed (`dot(vel + groundVel, dir)`, NOT `realized` â€” the sweep zeroes the realized into-component at the face; Jolt `HandleContact` likewise computes its impulse from `inVelocity`), with the **downward component cancelled** (Jolt's impulse down-cancel â€” a char landing on a box must not hammer it down at fall speed; gravity reaches the box through the solver). The shove goes to the backend via `setVelocity`. The soft solver contact alone caps the carry at ~0.4 m/s, so a faster char would stall; the transfer makes it keep up â€” no overtake, the kinematic-vs-dynamic limit Unity/Godot/Jolt solve with explicit push. Mass-independent, one-char-per-dynamic. Gates: oracle `full-speed push` + the gym `character` no-overtake. The "push into wall" pin squirts the box out â€” a bounded physical escape (the gym gate is no-explosion, not stays-pinned).
- **Dynamics block like statics â€” the sweep depenetrates EVERY contact (Jolt CharacterVirtual's model).** A walkable contact (static floor or box top) grounds + supports the char; a side contact blocks it at the face while the velocity transfer shoves the body. The earlier asymmetry (skip `bMass > 0 && !walkable`) was debt: it let the capsule penetrate a dynamic's side until the closest-point MTV flipped to the box's TOP face (which reads walkable â€” the test is only `gap < GROUND_SNAP && normal.y > maxSlopeCos`, and gap is always < snap while penetrating), ejecting the char upward by the full depth in one tick (walking into a crate teleported the char on top; jumping into a tall one tunneled). Depenetrating the side no longer kills the push because the transfer keys on the desired velocity. Carry + ground-snap stay static-only. Gates: oracle `stands ON a dynamic box` / `blocked at the face` / `slides off a tall box`; the gym `character` wall-stop + box-undisturbed stations.
- **Moving-platform carry.** A char on a translating/descending kinematic platform tracks it: the sweep scans the statics at `start` for the deepest walkable contact, reads that body's velocity (via `readBody`), and adds it to the motion. A true static reads velocity 0, so a flat floor never carries. The platform's velocity comes from `setKinematic(eid, pos, quat[, teleport])` â€” the runtime kinematic-pose API a moving platform / grab anchor calls each fixed step (`teleport` zeros the derived velocity for a jump that isn't motion). **Angular carry is a decline, not a gap**: no position-based reference cleanly carries a kinematic body's angular `dq` (AVBD freezes `mass â‰¤ 0` at `dq = 0`; XPBD's live-position re-eval is off AVBD's once-per-frame contact linearization). A powered rotating surface that must drive resting bodies is therefore a **dynamic** body (`mass > 0`) driven by a motor constraint, not a kinematic body. Gate: oracle `platform-carry`.

## Authoring â€” declarative components

Constraints and characters author as ECS entities, uploaded to the backend by the substrate's per-frame systems; a backend's imperative `setSprings` / `setJoints` still work. **Coexistence:** an empty authored constraint set hashes to the system's initial signature, so a component-free scene never uploads â€” use ONE path per constraint type (components OR imperative), not both.

- **`Spring` / `Joint` â€” a constraint is its OWN entity** holding two `@name` body refs (`a`/`b`) + local anchors (`rA`/`rB`), the consumer-shaped relation (`ecs.md`, like `Tween.target`). `Joint`'s angular lock authors `stiffness-ang: fixed` (âˆž, a `parse` trait hook); the default 0 is spherical. `ConstraintSystem` (fixed, `before: [StepSystem]`) uploads the derived `SpringDef` / `JointDef` to the backend. How a backend maps them to its own constraints (AVBD's soft joint, tumble's DistanceJoint/Spherical/Weld) is that backend's rule.
- **`Character` (`standard/character`) â€” the base kinematic controller** (`maxSlope` / `jumpSpeed` / `gravity` on a capsule `Body`). `CharacterPlugin` registers every `[Character, Body]` + exposes the eid-keyed drive `move` / `jump` / `pose`. **`Player` COMPOSES it** (requires `Character`, drives through it, adds look + camera) â€” controller tuning lives on `Character`, NOT `Player`.
- **Re-upload only on change** (an FNV signature over the authored set) gates the per-frame CPU adjacency rebuild. The signature folds each endpoint's `state.stamp`, so a same-update destroy+create realias of a referenced body forces the re-upload that rebinds the constraint on the new occupant (membership alone can't see it â€” ecs.md "An eid is a borrow"). Corollary: set a constraint/character's fields BEFORE its entity is first seen â€” a later edit re-uploads via the signature, but registration reads the fields once at first sight.
- **The drive API (`move` / `jump` / `pose` / `grounded`) is the `@dylanebert/shallot/character/core` subpath** â€” the eid-keyed drive + readback a custom controller uses (the happy path = the `Character` component + `CharacterPlugin` on the barrel). `move` / `jump` write the CPU drive maps, `pose` / `grounded` read the swept `CharState` â€” same-frame, no GPU buffer.
