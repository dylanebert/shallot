# Physics (AVBD)

`standard/physics` is a from-scratch GPU port of AVBD (Augmented Vertex Block Descent, Giles/Diaz/Yuksel SIGGRAPH 2025) over rigid boxes. The C++ reference is `reference/avbd-demo3d/`; the executable spec is the f64 CPU oracle in `tests/avbd/` (a faithful port of that C++). The GPU step (`step.ts`) and the SAT (`collide.ts`) reproduce the oracle. Build phases up the augmented-Lagrangian ladder; don't skip a layer.

## The oracle is the spec — validate against it, at the tier that's deterministic

- **`tests/avbd/` (CPU oracle) is the `bun run test:oracle` tier** (the `.oracle.ts` suffix — split out of the default `bun test` for speed since it runs hundreds of f64 frames; `bun run test:full` runs both and is the gate before a physics commit). It ports the C++ reference operation-for-operation (f64 where the C++ is f32). It's gold/closed-form validated (`oracle.oracle.ts`, `sat.oracle.ts`). When the GPU and the oracle disagree, the GPU is wrong until the oracle rules it out — the oracle is not the suspect.
- **GPU correctness is the gym `pile` scenario (real GPU).** Perf + the live demo live there too.
- **There is no software-adapter cross-check tier. The deterministic `bun test` tier is the f64 CPU oracle; the real-GPU gate is the gym.** `bun test` never binds a device: the bun-webgpu software adapter (Mesa lavapipe) is slow, flaky, and feature-incomplete (no usable subgroups/timestamp, the physics `build()` can time out), so a device-binding physics `bun test` is unreliable — the WSL bar went red on exactly those. Hardware-invariant logic (SAT feature keys vs gold, the incremental-greedy coloring, the penalty/dual/warmstart math) is validated deterministically against the oracle in `tests/avbd/*.oracle.ts` (CPU, f64). Everything whose truth needs the real device — the WGSL compile + execution, the subgroup-per-body solve, perf — is the gym `pile` scenario, on real hardware. Don't add a `*-gpu.test.ts` that binds a device.
- **Gate where it's deterministic, never on long-horizon trajectories.** Two float impls can't bit-match a stacking sim over hundreds of frames (Lyapunov divergence is physics, not a bug). The ladder: closed-form invariants (free-fall, resting `mg/k`-scale penetration); the **single-step-exact** gate (inject one identical frame-start state into GPU + oracle, step once each, compare to a derived single-step tolerance — no chaos accumulation, so it's tight, ~1e-7 pos in penalty mode); a statistical band (energy non-increasing, bounded penetration, no NaN) for chaotic long-horizon only. A single-step gate catches a math bug per-step that a trajectory gate would let hide.
- **f32 precision degrades with world-coordinate magnitude — test topology near the origin.** The contact gap `xA−xB` is a small difference of two large nearby coordinates (catastrophic cancellation; ulp ≈ |coord|·2⁻²³, ~1.4e-5 at coord 120), so absolute-coordinate magnitude sets the precision floor. Near-unstable topologies (tall chains, hard topples) destabilize as |coord| grows (band-breaching ~|coord|≈50, tunneling ~120 on lovelace at iters=10); well-conditioned piles tolerate it far out. A translate-invariant scene that changes behavior under a rigid world-offset is *this*, not a solver bug — diagnose by translating to the origin. **Don't make the oracle f32 to "match" the GPU**: the oracle's value is being *more* precise, so a disagreement tells you the true answer; an f32 oracle is just a second copy of the GPU that can't catch a shared bug. Large worlds (a long track) are the floating-origin / f64-storage problem, out of scope for the solver.
- **Two more f32 realities the robustness corpus surfaced — documented limits, not solver bugs.** (1) **Cross-scene contact-order non-determinism:** the contact pool is atomic-appended, so in a busy multi-body solve a near-unstable pile's f32 contact-force sum order varies run-to-run — enough to tip a hard topple into a tunnel or jitter a tall chain, even near the origin (each topology is deterministic *in isolation*). The fix (global deterministic contact ordering — stable-sort the contact array) is deferred to `backlog.md`; until then, test each topology in isolation. (2) **The f32 iteration floor:** the f64 oracle settles every topology at iters=10, but f32 needs more for the hardest — a 10-high free chain settles reliably only at iters≥~15 (iters≤12 under-converges it into a ~0.03–0.09 jitter), and a chaotic topple's settle *time* is iters-dependent. **Production ships iters=6** (the perf/robustness tradeoff — every fixed count explodes on *some* taller stack, so a known-harder pile bumps iters per-scene via `Physics.step.configure`; 6 settles a 10-storey stack — the collapse showcase wall — where 4 pancakes during the settle); the f64 oracle (`corpus.oracle.ts`) + the gym gates validate at **iters=10**, decoupled from the ship value (a production iters change never weakens them). **iters is a free tuning knob, NOT a correctness gate — the per-step math is GPU==oracle at *any* iters, so validate high and ship low.** Gates: the f64 oracle (`corpus.oracle.ts`) is the deterministic all-six band gate; the gym `pile` single-step-exact check covers all six topologies (GPU==oracle per-step, the tight correctness gate the static band lacked).

## Reference-fidelity — the port can't silently drift

Subtle points in the C++ that the oracle (and through it the GPU) must match exactly. Get one wrong and the GPU diverges from the oracle:

- **The narrowphase sphere test is the reference broadphase filter** (`dot(dp,dp) ≤ (rA+rB+SPECULATIVE_DISTANCE)²`, radius = `length(halfExtents)`) — it defines the contact set, so a pair can pass it and yield 0 SAT contacts. The BVH *prim* is the oriented-box world-AABB (`boxExtent`, tighter than the sphere bound — fewer candidates per body descent block, the memory lever) padded by `SPECULATIVE_DISTANCE`, a SAT-equivalent superset: any pair with real or speculative contacts has overlapping padded box-AABBs, so the box-AABB BVH + the sphere test + SAT produce the exact reference contact set (don't read the box-AABB prim as a contact-set change). The `+ SPECULATIVE_DISTANCE` pad is the broadphase half of the speculative band (next bullet); the velocity sweep (below) also pads the prim + sphere by the per-step velocity (`abs(vel)·dt` / `|vRel|·dt`) so a fast pair is found — see the velocity-sweep bullet.
- **Speculative contact band — moving-contact robustness, GPU == oracle == C++.** The SAT emits a contact while the boxes are still *separated* by up to `SPECULATIVE_DISTANCE = 4·COLLISION_MARGIN = 0.04` (Box2D's `4·slop`), carrying the true signed gap in `c0` (the clip vertex projects onto the reference face, so `c0.x = +gap + COLLISION_MARGIN > 0`). The solver is already the right shape — the normal force is repulsion-only (`force.x = min(force.x, 0)`), the unilateral speculative form (Firth 2011 / Box2D / Catto) — so this is a contact-*generation* change, **not** a solver change: a body within the band at frame start lands AT contact (no penetration pop, no tunnel) rather than overshooting through it. Two thresholds carry it, identical in `collide.cpp`, the f64 oracle, and `COLLIDE_WGSL`: `testAxis` aborts only past the band (`separation > SPECULATIVE_DISTANCE`), and the face clip keeps a vertex up to the band (`distance > SPECULATIVE_DISTANCE`). The broadphase pads to keep the pair visible — the C++/oracle sphere `r += SPECULATIVE_DISTANCE`, the GPU the same sphere test plus the **prim** AABB (not the query box) padded by it (combined slack = the band, the minimal find). **Keep `SPECULATIVE_DISTANCE` distinct from `COLLISION_MARGIN`** — the generation band vs the equilibrium offset, two magnitudes Box2D separates and webphysics conflates into one tuned literal; *derive* the 4× from Box2D, don't copy a flat literal. A penetrating vertex (`distance ≤ 0`) is unchanged, so settled-pile bands hold (the dual-layer rest shifts ≤ 2.3e-4 m — the box settles a hair above the bare margin, never below). The boundary this leaves open — a mover *farther* than the band at frame start (`v·dt` ≫ the band) tunnelling — is closed by the velocity sweep (next bullet). Gates: the oracle `speculative contact` single-step (red without the `testAxis` abort), the SAT gold `spec-face-*` configs, the gym `speculative-stop` (GPU == oracle on the real GPU, a 4-point speculative manifold).
- **Velocity sweep — anti-tunnel CCD, GPU == oracle == C++.** A body crossing the whole contact between frames (`v·dt` ≫ the static band) is caught at frame start by widening the SAT band along each axis to `max(SPECULATIVE_DISTANCE, closing)`, where `closing = max(0, dot(dRel, n))` and `dRel = (vₐ−v_b)·dt` is the relative displacement over the step (linear only; angular `ω×r·dt` neglected). Identical in `collide.cpp`, the f64 oracle, and `COLLIDE_WGSL`: `collide(…, dRel)` threads `dRel` into the `testAxis` abort (`separation > max(SPECULATIVE_DISTANCE, closing)`) and the face clip keeps a vertex to that same band. Like the speculative band it is a contact-*generation* change (the unilateral repulsion-only constraint then limits the approach), so no solver change — and it degenerates to the static band exactly at `dRel = 0`. **`max`, not `+`** is load-bearing: `SPECULATIVE_DISTANCE + closing` makes the band velocity-dependent even at settling speeds, feeding a body's own residual velocity back into the contact set and limit-cycling a marginal pile (the 8:1 `mixed-sizes` bridge); `max` keeps a slow body's band the velocity-independent static skin (closing ≪ skin), so only a genuinely fast mover (closing > skin) widens it. **Reference posture:** the broadphase velocity sweep *matches* webphysics (its tree fattens each AABB by `|vel|·dt`); the narrowphase swept-contact is a *motivated* deviation — webphysics keeps a fixed `contactSlop` narrowphase (no velocity), so a true fast mover tunnels through it; we adopt the Box2D/Bullet/Firth speculative-CCD completion (the sanctioned "battle-tested sub-technique outranks webphysics's cruder instance," cf. the Jolt reduce, the Box2D speculative band). The broadphase pads per-axis by `abs(vel)·dt` on prim + query (`SPECULATIVE_DISTANCE` stays prim-only — combined static slack unchanged) and the sphere by `|vRel|·dt`; *derive* from `v·dt`, never copy webphysics's hardcoded `1/60` horizon. Gates: the oracle `velocity sweep` single-step (red without the swept abort; a static body at the same gap stays separated, proving velocity-gating), the SAT gold `spec-swept-*` configs, the gym `swept-stop` (GPU == oracle, a fast box beyond the band caught at the surface).
- **`bodyA` is the higher creation index.** Broadphase orients each pair `a > b`; the SAT's A/B roles + feature keys depend on it. Dense body index tracks creation order for this reason.
- **SAT feature keys are bit-identical to the C++** (`collide.ts` packs them with u32 shifts) and stay the body-local clip-loop ordinal — never re-ordinal'd to storage rank. They key warmstart persistence; a wrong shift silently breaks it. Gated by `sat.oracle.ts` against the gold vectors on CPU; the GPU SAT reproduces it, checked by the gym single-step gate. **Don't re-ordinal to rank** — ours scans the slot by `(a,b)`+feature, so a rank renumber breaks the match (one key to two physical contacts when the reduced set shifts). It's one of several viable schemes (rank-indexed and position-proximity matching are the alternatives, each with a different shifted-key failure mode); `sat.oracle.ts` pins the current choice.
- **A face manifold is reduced to ≤4 points — a verbatim port of Jolt's `PruneContactPoints`** (`reference/jolt` ManifoldBetweenTwoFaces.cpp; the Gregorius GDC-2015 recipe; `reference/bullet3` b3ContactCache corroborates): plane-project, keep deepest-by-torque×depth, its farthest partner, then the furthest point on each side of that line (max quad area). It runs identically in `avbd-demo3d` collide.cpp (the harness fork), the f64 oracle, and the GPU WGSL, so `oracle == C++` and `GPU == oracle` hold; the reduction alone is gold-gated against an INDEPENDENT verbatim-Jolt extract (`reduce.oracle.ts` ← `reduce-gold.cpp`). It halves `CONTACTS_PER_PAIR` 8→4 (the memory win below). Reduction-only — no sort, no re-ordinal (see the feature-key bullet). avbd-demo3d's own SAT keeps up to 8; this is the one deliberate, cross-checked departure, motivated by the battle-tested engines, not webphysics.
- **Contact constraint:** `C = c0·(1−α) + Jᵀ·dq`, `c0 = basis·(xA−xB) + [COLLISION_MARGIN, 0, 0]`, `COLLISION_MARGIN = 0.01`. The margin shifts the resting equilibrium — a box settles ~`margin + mg/k` into contact, not bare `mg/k`. Don't "fix" that; it's the reference.
- **Velocity recovery is BDF1, and the angular diff is a 3-vector — not a 4-vector subtraction.** `v = (x − x_initial)/dt`; `vAng = qsub(quat, initialQuat)/dt` where `qsub(q, q0) = (q · inv(q0)).vec · 2` (the reference's overloaded quat `operator-`, **right**-multiplied). It pairs with the **left**-multiplied integrator `quat += dxAng = normalize(q + 0.5·δ⊗q)` (δ = `(ω, 0)` on the left). Diff right-multiplied, integrate left-multiplied — consistent for small angles; do NOT unify them. `x_initial` is the per-step pre-warmstart pose, saved after inertial init. Static/kinematic bodies (`mass ≤ 0`) skip primal + velocity.
- **Two different gravity terms.** The inertial target uses full gravity (`g·dt²`); the adaptive-warmstart start position uses `g·accelWeight·dt²` (`accelWeight = clamp(accel.y·sign(g)/|g|, 0, 1)`, needs `prevVelocity`). Don't conflate them.
- **Quat→mat3 on GPU uses the *active* form** (shallot's `composeTransform`), never the reference `maths.h rotation()` (passive transpose + dead code there). The oracle omits `rotation()` for this reason. Coordinates already match shallot: Y-up, quat `[x,y,z,w]`, right-handed, active rotation — a `Body.quat` feeds the transform firehose with no conversion.
- **6×6 solve is one cross-coupled LDLᵀ** (`maths.h solve()`), not two 3×3s.
- **The friction penalty-ramp gate reads the *pre-clamp* tangential magnitude, never the cone-clamped force.** `manifold.cpp:156` captures `frictionScale = ‖F_t‖` before the clamp; `:169` ramps the tangential penalty only when `frictionScale ≤ bounds` (sticking). A sliding contact has pre-clamp `‖F_t‖ > bounds` so it skips the ramp; gating on the *post*-clamp force (`length(force.yz)`, always `== bounds` once saturated) ramps a sliding contact's penalty every frame, over-stiffening the tangential solve until kinetic friction fades to ~0 (the box slides instead of decelerating at `μg`). So `manifold.ts contactForce` + the GPU `CONTACT_FORCE_WGSL` return the pre-clamp `frictionScale`+`bounds`; the dual must not recompute them from the clamped force. Energy-non-increasing + static-hold gates miss this — needs a sliding-scene parity gate (`oracle.oracle.ts` friction-settle, gym `friction-slide`).

## Rounded narrowphase — sphere + capsule

A new **producer** into the same type-tagged `pairContacts` seam, NOT a solver change — the solver consumes a rounded contact exactly like a box one. No AVBD/webphysics reference covers it (demo3d boxes only; webphysics has spheres, no capsule), so it's grounded in Jolt/Bullet's convex-radius idea.

- **Shape = a core inflated by a radius; one segment-segment closest-point function.** A sphere is a point + radius, a capsule a segment + radius. `closestSegments` (Ericson RTCD §5.1.9, the box edge-edge routine) between the two cores → subtract the radii → **one** contact (curved surfaces meet at a point — exact, not an approximation). A sphere is a zero-length segment, so sphere-sphere / sphere-capsule / capsule-capsule collapse to that one routine (`collideRounded`, `collide.ts` WGSL + `tests/avbd/rounded.ts`). The contact has the SAME `SatResult` / record shape as the SAT's (B→A basis in row 0, local arms, one feature key = `ROUND_FEATURE` 0x03000000). **No GJK/EPA** (analytic-closest-point), so GPU == oracle holds and the oracle isn't a second EPA impl masking a shared bug.
- **Rounded × polytope (box OR hull) = `collideRoundedPolytope`, one unified path** (`collide.ts` WGSL + `tests/avbd/rounded.ts`). A `Convex` is a scaled registered hull (a box = the built-in unit cube × half-extents; see "Convex hull narrowphase"), so a box and a hull share one routine. A **sphere** samples its centre → `closestPointOnPoly` → **1** contact (box: the `closestPointBox` clamp shortcut — geometrically exact; hull: the general `closestPointOnHull`, face-region → edge → vertex). A **capsule** is the **core segment clipped against the reference face** (`capsulePoly`) → up to **2** contacts sharing ONE reference-face normal — the richer capsule manifold. The contact normal is a mini-SAT between the core segment and the polytope, robust to an OVERHANGING capsule (its endpoints' closest points misfire; the SAT still finds the face it rests on — the mid-segment case endpoint sampling misses). **No hull-clip infra is GJK/EPA** (analytic clamp + SAT, exact). Oriented like `collideBoxBox` (A/B params in, basis B→A out), so the polytope-as-A ordering collides round-as-A on `−dRel` then swaps arms + flips the normal.
- **Dispatch by shape pair, in the collide pass.** `roundedA = (bShape == Sphere || bShape == Capsule)` (NOT `!= 0` — a hull is shape 3, a polytope). Both rounded → `collideRounded`; both polytope → `collideBoxBox` (both box) else `collideHull`; **mixed (one rounded, one polytope) → `collideRoundedPolytope`**. Gold/validation: closed-form (`rounded.oracle.ts`), the gym `pile` rounded narrowphase gate (production WGSL vs the f64 oracle `narrowphase`, byte-exact, run on the sphere/capsule rows), the gym `sat` hull kernel gate (rounded × hull configs incl. the capsule mid-segment overhang), and the live rounded rests in the gym `pile` `boundary: drum` (a static wall pen) — a flat-ground rounded pile rolls (no-tunnel only), a drum-held one settles on the box ground (the real rounded rest gate, what rounded-box unlocks).
- **Shape on `Body`: explicit `shape` (Box/Sphere/Capsule), `radius` folded into `halfExtents.w`.** The radius rides the authored `halfExtents.w` lane (the SoA shape-geometry grouping — core extents + radius read together; gpu.md "reuse over add"), so the pack/seed stays at the 10-binding floor with only one new `shape` slab binding (no seed split). The body cols-buffer gains a `B_ROUND` column = (bitcast kind, radius) — free (cols-buffer is one binding regardless of column count). The seed derives the per-shape **moment of inertia** (box / sphere `⅖mr²` / solid-capsule), and `boxExtent` + the sphere broadphase bound add the radius (a box has radius 0, so its path is bit-identical). Capsule core is local-Y (`size = (0, 2h, 0)`), oriented by quat.
- **A rounded contact MUST re-collide FRESH arms every frame — never the sticking-frozen ones.** The box-box warmstart keeps a sticking contact's frozen arms because a box feature key pins a persistent vertex/edge. Any pair **involving** a rounded shape (sphere/capsule, even vs a box) has a **constant** feature key that pins NO physical point — the closest point slides (a rolling sphere's contact, and its anchor on the box face, both move) — so freezing the arm anchors a stale point: any rotation spins the frozen arm into a tangential offset → torque → runaway spin → the normal constraint reads separated → zero normal force → the body sinks through. Gated `!(roundedA || roundedB)` (freeze ONLY box-box) in BOTH the GPU collide pass (the `if (oldLam.w > 0.5 && …)` arm-keep) and the oracle `manifold.ts initManifold` (`boxPair`). Symptom when violated: a sphere resting on a sphere (or rolling on a box) spins to ~10 rad/s and slowly sinks while the contact's `pen` ramps but `lam` stays 0 (the giveaway, read off `pairContacts`). The λ/penalty warmstart still carries — only the arms go fresh. Pinned by the *rolling* sphere-on-box gate (`rounded.oracle.ts`) — a settling body never rotates while it sticks, so frozen == fresh; the contact must be rotating to expose it.
- **A rounded contact stores the CORE arm; the solver re-applies the radius offset along the FIXED normal — never bake it into the stored arm (the rotational-stability rule).** The narrowphase (`collideRounded`/`collideRoundedPolytope`, oracle + WGSL) stores the local arm to the **core** feature (sphere centre / capsule segment point / box surface = its core at radius 0); `contactForce` AND the `c0` surface reconstruction (collide pass / `initManifold`) re-apply `∓radius·basis[0]` (`−rₐ·n` on A, `+r_b·n` on B). Baking the radius into the stored arm instead lets it rotate with the body's spin → a sphere's normal Jacobian `jAng·n = cross(−r·n, n)` goes nonzero → spin couples through the normal → a spun sphere tunnels (the same failure shape as the frozen-arm rule above, a distinct cause — fresh arms are necessary, not sufficient). The fixed-normal offset keeps `jAng·n = 0` (a sphere's normal force is torque-free). Box radius 0 ⇒ bit-identical to box-box. Pinned by the `rounded.oracle.ts` conservation invariants (frictionless spin is angularly transparent vs a free spin; rolling-without-slipping holds `|v| = ω·r`) + the gym `pile` rounded narrowphase gate (the core arm stored byte-exact vs the oracle — a baked-radius arm rotates with spin and would diverge here).
- **Residual ω decay on a sustained spin is the BDF1 quaternion integrator, NOT a contact bug — don't "fix" it.** A *free* (contactless) sphere or box bleeds a little `|ω|`/step (`qsub`/`qadd` are only small-angle consistent), identical in-contact — so the conservation tests compare the in-contact spin to a free-spin reference (the contact must be *transparent*, not zero-damping). Don't deviate the friction Hessian to remove it: the reference (`manifold.cpp:90,117`) stamps the full penalty `K=diagonal(penalty.xyz)` regardless of the cone (a μ=0 tangent Hessian adds drag with no force) — the canonical AVBD formulation; changing it breaks GPU == oracle == C++.

## Convex hull narrowphase

Another producer into the same `pairContacts` seam — the box-box SAT generalized to arbitrary faces + edges. No AVBD/webphysics reference covers it (demo3d boxes only); grounded in Jolt + Bullet's GPU SAT. **No GJK/EPA** (EPA's dynamic allocation + data-dependent iteration is GPU-hostile, the class that sank the caramel RT trace; Bullet's own GPU path drops it for SAT). The oracle is `tests/avbd/hull.ts`; the WGSL is `HULL_WGSL` in `collide.ts`.

- **One `Convex` type, box OR hull — scale-unified.** `collideHull(a, b)` takes two `Convex`es; each is a registered hull (`hr`) posed + scaled. **A box is the built-in unit cube (`UNIT_CUBE_ID`) scaled by its half-extents** (vert × scale, face normal × 1/scale renormalized, edge × scale), so **box × hull and hull × hull share one branch-free path**, exactly like the oracle feeding a box as its `boxHull`. **box × box stays on `collideBoxBox`** (its analytic Obb projection + warmstart-critical feature keys are validated separately; the oracle's `narrowphase` routes box × box → `collide`, so the GPU must too). The SAT search is face-A + face-B normals + edge-A × edge-B cross products, the 0.95/0.01 edge/face bias, the reference-face Sutherland-Hodgman clip, and the **shared `pruneContacts` reduce-to-4** (the within-band candidate set is ≤ `MAX_CANDIDATES` = 8 — a full cone octagon — so it reuses the box path's array bound, no wider duplicate). Hull feature keys are `HULL_FACE`/`HULL_EDGE`-tagged (distinct from box AXIS_FACE/EDGE + ROUND), body-fixed clip ordinals (same warmstart discipline). Frozen-arm warmstart is **polytope × polytope** (box/hull pin a clip vertex; rounded shapes stay fresh-armed) — `!(roundedA || roundedB)`.
- **Geometry: the `Hulls` registry → `packHulls` → the `hullData` binding.** `Hulls` (a `Registry<Hull>`, like `Meshes`) holds verts / polygonal faces (normal + offset + CCW vert indices) / unique edge dirs. `packHulls` serializes all registered hulls into one flat `array<u32>` (a header table indexed by `hullId`, then each hull's verts/faces/faceIdx/edges; `hull.ts` `HULL_HEADER`/`HULL_FACE_STRIDE`). The plugin re-uploads (`step.setHulls`) only when the registry size changes (hulls are static); a 1-u32 stub until then. The collide pass binds it at **binding 6** (7 of the 10-buffer floor). A body's `hullId` rides `halfExtents.w` (a hull has radius 0, so the lane is free; `halfExtents.xyz` carries the AABB half for mass + broadphase) — **no new pack-pass binding** (the pack is already at the 10-floor). The seed extracts `hullId = u32(srcHalf.w)` for shape 3, stores it in `B_ROUND.z`.
- **Hull inertia = the AABB box approximation (interim).** A hull body's moment is `boxMoment(AABB)` — exact for a box-hull (every hull tested), an over-estimate for tet/cone. The settled approach (exact polyhedral integration baked to the COM-centered principal-axis frame, computed once at registration — Jolt's signed-tetrahedron covariance) lands when a non-box-hull *dynamics* test needs it (rest scenes use box-hulls).
- **Compile discipline — the narrowphase is FOUR pipelines, one per shape-pair class (gpu.md "DXC shader compilation": dead code isn't free, optimize via pipeline splits).** DXC compile is superlinear in kernel size, so a monolithic box+rounded+hull kernel paid the full cost. `collide.ts` exports the chunks `HELPERS_WGSL` (shared math/structs/manifold primitives) + `BOXBOX_WGSL` + `ROUNDED_WGSL` + the three hull chunks `HULL_CORE_WGSL` (the shared scale-unified polytope substrate — `Convex` + accessors + `projectPoly` + `bestPolyFace`) + `HULL_SAT_WGSL` (`collideHull`, polytope×polytope) + `ROUNDED_POLY_WGSL` (`collideRoundedPolytope`, rounded×polytope), recomposed into `COLLIDE_WGSL`/`HULL_WGSL` for the narrowphase gates (the gym `pile` rounded gate + the gym `sat` gate); `step.ts`'s `collidePass(chunk, gate, call, ownsLifecycle)` + `buildCollide` build **box** (`HELPERS+BOXBOX`) / **rounded** (`HELPERS+ROUNDED`) / **hull** (`HELPERS+HULL_CORE+HULL_SAT`) / **rounded-poly** (`HELPERS+HULL_CORE+ROUNDED_POLY`), so a box-only pipeline never compiles a hull/rounded SAT. The four `createComputePipelineAsync` run concurrently, so wall-clock compile ≈ the largest chunk — `collideRoundedPolytope` ~600 ms (the capsule segment-clip + closest-point, the pole), above box ~515 ms and `collideHull` ~360 ms (min-over-3, nvidia/lovelace DXC). **If startup compile needs cutting further, split `capsulePoly` into its own pipeline (a 5-way) — drops the pole to box ~515 ms — or drop capsule.** **Routing:** all four dispatch over the SAME live pair slots (indirect off `pairArgs`) and act only on their class (the gates are mutually exclusive + exhaustive: box×box / rounded×rounded / polytope×polytope-not-both-box / exactly-one-rounded); the box pipeline OWNS slot lifecycle (clears every INVALID/separated slot, the sphere filter being shape-aware), the other three fill/clear only their own class, so each slot is written once — 3 extra cheap early-out dispatches/step (collide runs once per fixed step), no partition pass. Within each kernel, the scale-unified `Convex` keeps box AND hull accessors branch-free and `projectPoly` rotates the SAT axis into local frame ONCE (`dot(pos+R·v,axis)=dot(pos,axis)+dot(v,Rᵀ·axis)`, a math identity + runtime win); all Metal-safe (no new dynamically-indexed private arrays — the box-SAT spill trap). **REFUTED — don't re-attempt:** constant loop bounds + dynamic break (DXC *unrolls* them at bounds ≤16 → 3–10 s; dynamic bounds stay rolled, fine); precomputing world verts (Metal-array risk); splitting `collideHull`/`capsulePoly` *per shape within ONE kernel* (duplicates the SAT body — distinct from the pipeline split, which puts each whole SAT in its OWN kernel). **The cost is DXC-on-Windows only** (Linux/Tint compiles the same kernel in ~2 ms — don't measure the split there), and the metric is noisy (±1.5 s, cold-cache outliers to ~10 s): min-over-3 on the gym `phys-collide-*` compile spans, not single bench runs.
- **Gold/validation: two independent gates.** (1) **box-as-hull == box-box `collide`** — a box IS a convex hull, so `collideHull` fed box-hulls reproduces the box-box oracle on every face manifold (normal + world points + gaps, exact); `collide` is itself avbd-demo3d-gold-gated, so this transitively validates the whole generalization (axis search → clip → reduce) on boxes. The strongest gate. (2) **`bullet3-sat-harness`** (`gen-hull-gold.ts` → `hull-gold-vectors.json`, tet/cone) for genuinely non-box geometry — gated on the **separating axis only** (Bullet's points/depths are reference-convention-dependent; the axis is the one reference-independent invariant). `hull.oracle.ts` (CPU oracle), `hull-pack.oracle.ts` (the packing round-trip), the gym `sat` hull kernel gate (production WGSL == oracle over box-as-hull / tet / cone / box × hull / rounded × hull) + the full-pipeline rests (box-hull on box ground, two box-hulls stacking, box-hull on box-hull ground). Hull bodies ride the gym `pile` scenario as a perf + no-tunnel cell; the gym `sat` kernel gate is their tight validation (byte-exact codegen, not a watchable scene).

## Kinematic character

The character is a `mass ≤ 0` capsule `Body` whose pose a CONTROLLER owns, NOT the solver. The f64 spec is `tests/avbd/character.ts` (the executable spec the runtime CPU sweep reproduces, exactly as the oracle is the spec for the solver); webphysics has no character. No AVBD reference covers it — it's the standard kinematic-controller recipe (Unity/Godot/Jolt) grounded on the rounded/hull closest-point narrowphase.

**The runtime controller is the CPU sweep (`standard/character`) — there is no GPU character pass.** `CharacterSweepSystem` runs `sweepCharacter` (`standard/character/sweep.ts` — the f32-tier twin of the oracle, validated against it by `character-sweep.oracle.ts`) for every `[Character, Body]` each fixed tick in the FIXED group, `before: [StepSystem]`, so it lands BEFORE the GPU solve. The result is the input→pose→camera latency the pillar targets: a same-frame CPU path with no GPU readback (mouse-look already was; position now joins it). The camera follow snapshots `pose` off the CPU `CharState` in `PlayerSnapshotSystem` (`standard/player`, fixed, `after: [CharacterSweepSystem]`) so it captures THIS tick's swept pose, then renders `lerp(prev, curr, fixedAlpha)` — the only camera lag is the kept one-tick interpolation + the irreducible display fence, no readback. The coupling is **one-way**: the sweep writes the character's fresh pose into the GPU `bodies` buffer as a kinematic body (`PhysicsStep.setKinematic(eid, pos, quat, false, realizedVel)` — the moving-platform mechanism; the explicit `realizedVel` excludes the cosmetic ground snap so the carry-of-riders + broadphase pad read the swept motion), so GPU dynamics collide against the CURRENT-tick player; the sweep itself reads the static collision world from the authored `Body` slab and dynamic / moving-platform candidates one fixed tick stale off a `Mirror` of the `bodies` firehose (a frame-old dynamic pose is fine for the sweep). The push is the **full-CPU apply (variant A)**: a shoved dynamic's new velocity is written straight to its GPU `B_VELL` via `PhysicsStep.setVelocity` (which also wakes it), no GPU character work. The push-apply A/B is **resolved → ship variant A**: the CPU sweep's push impulse matches the f64 oracle (the `character-sweep.oracle.ts` push parity, negligible divergence) and the real-GPU push trajectory (the gym `character` no-overtake) tracks the fresh-velocity (≡ variant B) path. The thin-GPU-additive apply (variant B) is **unbuilt** — the A-vs-B difference is only the perpendicular component of the pushed body's velocity staleness (the along-push component clamps to the desired speed in both), ≈ 0 for a clean push, so it isn't worth a GPU pass; don't re-propose it without a measured pile-push artifact. The drive surface is `move`/`jump`/`pose`/`grounded` (`character/core`), reading/writing the CPU `CharState`.

**Validation tiers — no GPU char pass to single-step compare against.** The sweep's own correctness is the deterministic CPU oracle: `character-sweep.oracle.ts` (the f32 sweep == f64 oracle — slide / step-up / coyote / jump / slope / push / carry / cull-equivalence) + `character.oracle.ts` (the oracle's own feel + conservation tests). The GPU **coupling** (`setKinematic` uploads the pose, the solver then pushes the box / carries riders) is the real-GPU gym `character` live scene — drop-to-rest, wall-stop + box-undisturbed, full-speed push no-overtake, push-into-wall bounded, slope hold-vs-slide. The input→camera latency is the gym `character` `probe=1` gate (`scripts/physics-bench.ts` `character probe` row): under a self-calibrated GPU-load knob it asserts the position pose updates same-frame as orientation, no readback in the path. The former seeded single-step GPU==oracle gates were redundant once the sweep is the sole runtime path — its per-step parity is the CPU oracle, not a second GPU copy.

The bullets below describe the controller algorithm, shared by the CPU sweep (`sweep.ts`) and the f64 oracle (`character.ts`):

- **The all-static dual-ramp gate (a SOLVER feature the sweep relies on).** `updateDual` skips an all-static manifold (`m.a.mass ≤ 0 && m.b.mass ≤ 0` → early return): a contact no dynamic body can resolve is never satisfiable by the primal, so the reference's unconditional ramp (`solver.cpp:230` runs every force's `updateDual`) would escalate its penalty unbounded (a released endpoint later gets yanked by the accumulated λ). REQUIRED in the oracle (its O(n²) broadphase generates the kin-static contact); the GPU dual pass carries the same gate for parity + defense, though the GPU broadphase never generates a kin-static contact (next bullet) so it's inert there. Gated red-first + closed-form in `oracle.oracle.ts` / `character.oracle.ts`.
- **The character owns no broadphase blocks** (it's `mass ≤ 0`, classed static for ownership), so a dynamic partner owns every dyn-character pair → **the character pushes dynamics** (the dynamic's primal resolves the overlap) and **carries bodies resting on it** (the contact + its realized velocity), while kin-static / kin-kin contacts are never generated on the GPU — **the sweep, not the solver, stops the character at a wall**. So the GPU needs NO broadphase/inertial change: the sweep's `setKinematic` overwrites the char's `B_POS`/`B_VELL` before the solve, and the all-static dual gate keeps any incidental kin-static contact inert.
- **The controller is collide-and-slide (move-then-depenetrate), and its push-out normal is the geometric CLOSEST-POINT MTV, not the solver's SAT reference-face normal.** The MTV is the closest point between the capsule CORE segment and the polytope (`closestPointOnHull` ↔ closest-on-segment alternation). At a step edge the closest-point normal tilts UP, so the rounded bottom climbs a sub-radius step for free; the SAT face normal would wedge the capsule horizontally. This is why the character builds on the rounded/hull narrowphase yet uses the closest-point primitive, not the manifold SAT.
- **Three controller subtleties that are decisions, not tuning:** (1) gravity integrates ONLY while airborne (gated on last tick's `grounded`) — applying it every tick leaves a tangential residual after the perpendicular push-out, so the capsule creeps even on flat-enough ground; gating it holds a walkable slope and lets a too-steep one slide. The gravity is **per-character** (`Character.gravity`, default 0 = the configured world gravity, which the sweep reads via `PhysicsStep.gravity`) — a player falls/jumps snappier (e.g. `−50`) than the rigid bodies without touching the world's `−10`. (2) ground-snap pulls a grounded capsule onto the surface (gap 0, glued on stairs/slopes) but is SKIPPED while rising (`realized.y > 0`) or it pulls the capsule back off a step it's climbing. (3) `grounded` keys on a walkable contact within a snap band, so a settled capsule reads grounded at gap 0 without flickering. The realized velocity (`(sweptPos − start)/dt`) feeds the AVBD broadphase pad + the carry; the snap is excluded from it.

- **The sweep is culled — the oracle `gather` verbatim, on the CPU.** `gather` (`sweep.ts`) keeps only bodies whose bounding sphere can reach the capsule within this tick's motion budget (a derived contact-set-preserving sphere superset: every phase gates on `gap < GROUND_SNAP` or `depth > 0`, so a culled body is output-neutral), so the cost is independent of world body count — the gym `character` `filler` sweep piles inert far statics past every station's reach and the gates stay green. Candidates keep scan order — the order-dependent selections (first-max depth, last-walkable groundNormal) are bit-identical to a full scan. The carry's `groundVel` re-gathers with the full band (the provisional gather assumes `groundVel = 0`). Loud guards (`SweepDiag`): `overflow` (the per-character candidate cap `MAX_CHAR_CANDIDATES` = 64 — first 64 in scan order kept) + `guard` (travel exceeded the band's budget — the spawn-inside class). The `cull: false` brute seam is bit-identical (the cull-equivalence tests pin culled == brute EXACT, f64). A cached-plane (Jolt `SolveConstraints`-style) slide was evaluated and REJECTED: the free step-up depends on RE-PROBING at the penetrated pose (the closest-point bullet above) — a start-pose cached plane pushes the char straight back, and Jolt only climbs stairs via a separate `WalkStairs` it layers on top; don't re-propose plane caching without it.
- **Jump = coyote + buffer, single jump.** A jump fires only when BOTH the coyote credit (refilled to `COYOTE_TIME` while grounded, decays airborne) and the jump buffer (set by `jump()`, decays) are positive, and CONSUMES both — so a held/spammed button can't re-fire mid-air. Pinned by oracle tests (spam → one jump per landing; buffer fires on landing; coyote just after a ledge) + the gym `character` jumper station (real-GPU spam → one jump per landing).
- **Full-speed push = velocity transfer.** Each tick, for every dynamic the char is in **contact** with (gap within the `GROUND_SNAP` touch band — the sweep leaves the char AT the face, so penetration is never the trigger), the push drives that body's velocity along the push normal (char → dynamic) up to the char's **desired** into-speed (`dot(vel + groundVel, dir)`, NOT `realized` — the sweep zeroes the realized into-component at the face; Jolt `HandleContact` likewise computes its impulse from `inVelocity` relative to the body), with the **downward component of the transfer cancelled** (Jolt's impulse down-cancel — a char landing on a box must not hammer it down at fall speed; gravity reaches the box through the solver). The shove is written to the GPU `B_VELL` via `setVelocity` (variant A); the solver's inertial pass integrates it forward. The soft solver contact alone caps the carry at ~0.4 m/s, so a faster char would stall against the box; the transfer makes it keep up — no overtake, the kinematic-vs-dynamic limit Unity/Godot/Jolt solve with explicit push. Mass-independent (the kinematic char overpowers; a max push-force cap is a later refinement) and one-char-per-dynamic (two chars shoving one box would race). Gates: oracle `full-speed push` (char ends BEHIND a box it shoved forward) + the gym `character` no-overtake check (real GPU). The "push into wall" pin (a capsule driving a box into an immovable wall) squirts the box out — a bounded, physical escape (the gym `character` push-into-wall gate is no-explosion, not stays-pinned).
- **Dynamics block like statics — the sweep depenetrates EVERY contact (Jolt CharacterVirtual's model).** Statics and dynamics are both contact planes in the collide-and-slide loop: a walkable contact (static floor or a box top) grounds + supports the char; a side contact blocks it at the face while the velocity transfer shoves the body. The earlier asymmetry (skip `bMass > 0 && !walkable`, the sink-into-a-box fix) was debt: it let the capsule penetrate a dynamic's side until the closest-point MTV flipped to the box's TOP face — which reads walkable, since the test is only `gap < GROUND_SNAP && normal.y > maxSlopeCos` and gap is always < snap while penetrating — and the sweep then ejected the char upward by the full depth in one tick. Walking into a chest-high crate teleported the char on top; jumping into a tall one tunneled through. Depenetrating the side no longer kills the push because the transfer keys on the desired velocity (bullet above). Carry + ground-snap stay static-only. Gates: oracle `stands ON a dynamic box`, `walking into a dynamic box's side is blocked at the face`, `jumping into a tall dynamic box's side slides off`; the gym `character` live wall-stop + box-undisturbed stations.
- **Moving-platform carry.** A char on a translating/descending kinematic platform tracks it: the sweep scans the statics at `start` for the deepest walkable contact (the ground it rests on), reads that body's velocity (Mirror-read off `B_VELL`), and adds it to the motion. A true static reads velocity 0, so a flat floor never carries. The platform's velocity comes from `PhysicsStep.setKinematic(eid, pos, quat[, teleport])` — the runtime kinematic-pose API a moving platform / grab anchor calls each fixed step; it writes `B_POS`/`B_QUAT` + derives `B_VELL` from the per-step pose delta (`teleport` zeros that velocity for a jump that isn't motion). Angular carry is not tracked (a spinning platform carries by its COM velocity only), and this is a **decline, not a gap**: no position-based reference cleanly carries a kinematic body's angular `dq` (AVBD freezes `mass ≤ 0` at `dq = 0`; XPBD's live-position re-eval is off AVBD's once-per-frame contact linearization). A powered rotating surface that must drive resting bodies is therefore a **dynamic** body (`mass > 0`) driven by a motor constraint, not a kinematic body. Gate: oracle `platform-carry` (horizontal track + descending stays-grounded).

## Authoring — declarative components

The constraint sets author as ECS entities, uploaded to `PhysicsStep` by per-frame systems; the imperative `setSprings`/`setJoints` still work. Characters author ONLY as `[Character, Body]` components (the CPU sweep drives them via `character/core`); there is no imperative character API. **Coexistence:** an empty authored constraint set hashes to the system's initial signature, so a component-free scene never uploads — use ONE path per constraint type (components OR imperative), not both.

- **`Spring` / `Joint` (`standard/physics`) — a constraint is its OWN entity** holding two `@name` body refs (`a`/`b`) + local anchors (`rA`/`rB`), the consumer-shaped relation (`ecs.md`, like `Tween.target`). `Joint`'s angular lock authors `stiffness-ang: fixed` (∞, a `parse` trait hook); the default 0 is spherical. `ConstraintSystem` (fixed, `before: [StepSystem]`) uploads.
- **`Character` (`standard/character`) — the base kinematic controller** (`maxSlope`/`jumpSpeed`/`gravity` on a capsule `Body`). `CharacterPlugin` registers every `[Character, Body]` + exposes the eid-keyed drive `move`/`jump`/`pose`. **`Player` COMPOSES it** (requires `Character`, drives through it, adds look + camera) — controller tuning lives on `Character`, NOT `Player`.
- **Re-upload only on change** (an FNV signature over the authored set) — gates the per-frame CPU adjacency rebuild. `setJoints` itself preserves an UNCHANGED slot's live record (warmstart λ/penalty, active flag, a `setJointAnchor`-moved world anchor): only changed/new slots get a fresh record + the construction guards. Load-bearing for re-authoring under load — a full re-init re-judges a loaded chain's stretched pins at the reach guard (disconnecting it) and zeroes every λ (a visible release drop); the grab pattern (append/remove a joint at the TAIL of the same defs) relies on it. Gated by the gym `constraints` re-author check. Corollary: set a constraint/character's fields BEFORE its entity is first seen (a later edit re-uploads via the signature, but registration reads the fields once at first sight). The change-gate matches `lastHullCount`.
- **The drive API (`move`/`jump`/`pose`/`grounded`) is the `@dylanebert/shallot/character/core` subpath** — the eid-keyed drive + readback the gym `character` scenario + a custom controller use (the happy path = the `Character` component + `CharacterPlugin` on the barrel). `move`/`jump` write the CPU drive maps, `pose`/`grounded` read the swept `CharState` — same-frame, no GPU buffer.

## Storage + the Body / Transform contract

- **`Body.excludes [Transform]`.** Each is single-writer in CPU space. `Body` carries the spawn pose (pos/quat) + mass properties (halfExtents/mass/friction) on slab; physics owns the GPU pose after spawn. `mass ≤ 0` = static. Don't let `Transform` double as both user-author and physics-output surface (bulk readback into a slab the user also writes is a corruption hazard).
- **Solver state is eid-indexed over `capacity`, persistent across frames.** `bodies` / `colors` / `colorScratch` index `[col*capacity + eid]`; a body's state lives at its eid slot and survives spawn/despawn. **`maxBodies` = `capacity`** (the live + eid + dispatch bound — a scene runs up to `capacity` bodies). The `pairList` + `pairContacts` + CSR pools size by the **eid space × `PAIRS_PER_BODY`** (each body owns a fixed pair block at `eid·PAIRS_PER_BODY`, the warmstart bullet below). The `budget` arg (`PhysicsStep.create`) is **vestigial** — the store sizes to `eidCap`, not a separate pool — kept only for call-site compat. A body that overlaps more than `PAIRS_PER_BODY` neighbors trips the loud `counters[3]` (the descent's nearest-K prune drops the farthest, never silently) — there is NO global pool, so `counters[4]` is dead (a body can't exceed its own block); `counters[5]` is the body-pool (live count > `capacity`). `PhysicsStep.bytes` reports the total (`pairContacts` dominates — see the warmstart bullet). A single per-frame **GPU `pack`** (`PhysicsStep.pack`, the firehose) scans `0..capacity` gated on the Body membership bit and does two things with **no CPU per-entity iteration**: (1) compacts the live eids into the dense→eid map `eids` (`eids[0]` = live count, `eids[1+d]` = the d-th live eid, read by every body pass as `i = eids[1+gid.x]`); (2) one-time-**seeds** a new body's eid slot from its authored slabs (moment derived GPU-side, velocities zeroed) gated on a GPU `seeded` flag — existing bodies' slots are untouched (a mid-sim spawn never disturbs the pile), and a non-member eid resets its flag so a recycled eid re-seeds. **The pack is an eid-SORTED MULTI-workgroup count→scan→scatter compaction (C1.3)** — one lane per eid, deterministic over the membership bitset, NOT an arbitrary atomic-append. The prior single-WG form had each lane serially walk `capacity/256` eids — an O(capacity) span on one SM (~130 µs/frame at the default 65536 capacity regardless of live count); the multi-WG form is flat ~8-9 µs at any capacity, so a default-capacity app stops paying it. The scan's lane 0 also clamps + publishes (the prior standalone `clampCount` pass, fused — deletes a phase boundary): bounds the count to `capacity` (loud overflow → `counters[5]`), publishes the BVH prim count + the body-pass dispatch args (`dispatchArgs` = `ceil(count/64)`) + the per-eid-block-pass dispatch args (`pairArgs` = `ceil(count·PAIRS_PER_BODY/64)`). Body passes (aabb / broadphase / inertial / coloring / primal / velocity / compose) dispatch **indirect** off `dispatchArgs`; the per-eid-block passes (collide / dual / CSR) dispatch **indirect** off `pairArgs` (one lane per live-body × `PAIRS_PER_BODY` slot, lane → d → owner eid → slot). The standalone gates (no `packGate`) set the map + count + both arg buffers directly via `gateSetCount` (identity map, already sorted) and seed `bodies` by `writeBuffer`.
- **A `Body`+`Part` renders at the physics-owned pose — physics composes into the firehose, no `Transform`.** `BodyComposeSystem` (`after: [BeginFrameSystem]`, `before: [PrepassSystem]`) scatters the pose into `transforms[eids[1+d]]` (`PhysicsStep.compose`), after the Transform compose wrote that slot stale and before the renderer reads geometry. Scale is 2·halfExtents (the cube mesh is unit) — render-only; the body pose is scale-free. `Part`'s `requires: [Transform]` is advisory (only `excludes` throws), so a `Body` carries `Part` directly. **The eid-indexed `Body.pos` CPU slab is NOT populated per frame** — a CPU consumer reads the live pose by Mirror-ing `Physics.step.bodies` (eid-indexed). **The composed pose is render-interpolated:** `compose(encoder, transforms, alpha = time.fixedAlpha)` blends prev→curr pose (lerp pos, nlerp quat shortest-arc) so a >60Hz render doesn't repeat a fixed-step pose then jump. The prev pose needs **no extra column or snapshot pass** — the inertial pass already saves `x⁻` (last frame's settled pose) into `B_INITL`/`B_INITQ` before warmstart mutates `B_POS`, so prev = `bInit*`, curr = `bPos`/`bQuat` (equal for a static/freshly-seeded body → no-op; `alpha = 1` = the bare current pose). **`B_INITL` is load-bearing for the contact constraint** (`dq = bPos − bInitL`, so a static MUST read `bInitL == bPos` for `dq = 0`) — do NOT repurpose it as a render-prev for kinematic-mover interpolation (deriving `B_INITL = pos − vel·dt` makes a moving char's `dq` non-zero → the contact over-pushes; a kinematic platform therefore renders at the fixed rate, no sub-step interp). Solver gates read raw `bodies`; the compose is pinned by the gym `interpolation` gate (the settled-pile firehose-pose can't be — prev ≈ curr there).
- **The contact buffer is a source-agnostic, type-tagged producer seam.** Generation (the LBVH broadphase descent → per-eid pair blocks → the narrowphase SAT → in-place manifold writes) and solve are separate passes joined by the `pairContacts` buffer. The solve branches on the type tag (`kind`: 0 = inactive/cleared, 1 = `CONSTRAINT_CONTACT`; a zeroed record reads as inactive, so `clearBuffer` is a valid reset). A later non-body contact source (voxel/heightfield/SDF) drops in as a pre-solve producer that writes contact records — not a new shape-pair, not a solver change. Don't fuse SAT into the solve kernel.
- **Warmstart is in place in ONE persistent `pairContacts` store, keyed by the owner eid's FIXED per-body block slot (no hash, no compaction, no separate store, no cache pass).** Each frame the broadphase descends once and writes each live body's fixed block `pairList[eid·PAIRS_PER_BODY …]` directly (cheap `vec2` pairs, the body's overlapping neighbors **insertion-sorted by neighbor dense slot** = eid order; unused slots cleared to `INVALID`). A pair at slot `s = ownerEid·PAIRS_PER_BODY + k` carries its manifold at `s*CONTACTS_PER_PAIR` in `pairContacts` (SoA, `pairContacts[col*recordCap + rec]`) — the SAME records this frame's contacts write to AND last frame's λ/k persist in. The collide (dispatched over `liveCount·PAIRS_PER_BODY` lanes, lane → d → owner eid → slot) matches each new contact by scanning that slot's prev records (read before any write) for the SAME pair `(a, b)` + feature key, carries the decayed λ/k (+ sticking arms) in place, then overwrites the slot's records; the dual writes the final λ/k back, so next frame's collide warmstarts off them directly. **The `(a, b)` pair-identity gate replaces the hash** — a slot reused by a different pair fails it → cold-start, no probe, no separate table. A separated / 0-contact / `INVALID` slot clears its records (kind 0). **The slot base is the owner's eid alone**, so it is STABLE across frames unless that body's own candidate set flickers — **local** warmstart fragility (webphysics `broadPhase.ts` `bodyBase = body·pairsPerBody`), the precondition the in-place warmstart needs. A prefix-sum-compacted slot was tried and replaced: its slot was a global offset → ANY body's overlap-count change shifted every downstream pair's slot → the `(a,b)` match missed → total warmstart collapse on a churning pile → the tall-dense-pile explosion (measured: 16384 l12 0.7% → 99.2% warmstart, 195 → 0.6 m/s; 32768 l4 churn 9.2% → 99.9%). Loud guards: `counters[3]` per-body block overflow (a body with > `PAIRS_PER_BODY` neighbors — keeps the **nearest-K by center-dist²** + pins statics, drops the farthest; 0 in a normal scene, a small graceful count at stress), `counters[5]` body-pool overflow. `counters[4]` is **dead** — there is no global pool (a body can't exceed its own fixed block), so the compacted-pool overflow that the prefix-sum model could trip can't happen. `counters[6]` counts warmstarted contacts; `counters[7]` counts static-support pairs dropped DESPITE the pin (only when a body has > `PAIRS_PER_BODY` *static* neighbors — a box in a static cage; 0 in real scenes). **Importance prune + static-pin:** on overflow the descent keeps the **nearest-K by center-dist²** (webphysics `broadPhase.ts`) and pins statics (`d2 = −1`, never the farthest → never evicted), so the dropped candidate is always the least-important — not an arbitrary BVH-traversal-order drop. The prune+pin's symptom (a dropped support → fall-through) is caught by the bench fall-through guard (`scripts/physics-bench.ts`, keyed on the live-band `throughFloor` + `counters[7]`). **Caveat:** center-dist² mis-ranks a fast far approacher (large center-dist² but a real swept contact) — fine for settling/dense piles, flag if it fights the velocity sweep for genuine fast movers. **Why a per-eid FIXED block, not the prefix-sum compaction:** the compaction's memory win over the *original* per-body block was the 8/body vs 16/body slot count, NOT the compaction itself — so a per-eid block at `PAIRS_PER_BODY = COMPACT_PAIRS_PER_BODY = 8` is the SAME memory AND pair-stable. **Memory:** `pairContacts` = `eidCap · PAIRS_PER_BODY · CONTACTS_PER_PAIR · CONTACT_VEC4 · 16 B`, `CONTACTS_PER_PAIR = 4` (the Jolt reduce-to-4): ≈235 MB at `capacity 65536` (`scripts/physics-bench.ts` reports the total store size via `PhysicsStep.bytes`; ~265 MB with the rest of the step's buffers). `checkContactStore(eidCap, …)` guards the device per-binding limit, and `acquireDevice` (`engine/runtime/gpu.ts`) requests the adapter's full `maxStorageBufferBindingSize` + `maxBufferSize` (desktop + Deck expose ~2 GB), so the 65536 binding allocates on ~all devices; over-provisioning fails loud + clear via `checkContactStore` (`UnsupportedError` naming the buffer + needed-vs-available + remedy), pinned by `tests/avbd/storage.oracle.ts` (no device); the real-GPU bind at full cap is exercised by the bench's high-count sweep cells. Correctness gates: the gym `stack-warmstart` (3-box stack, GPU == oracle, exact 12/12 carry — the `merged == total` arm proves ≈100% persistence on a settled pile) + the live band; the overflow counters (`counters[3]`/`[5]`/`[7]`) route to the bench's fall-through guard.
- **The broadphase is two regimes behind one block contract.** `record` keys on the frame-stale live count (the `boundBodies` readback word; `gateSetCount` sets it directly): ≤ `StepParams.smallN` (default `SMALL_N` = 1024; 0 = always BVH — the gym pile `smallN` param is the A/B lever) runs ONE LDS-tiled O(n²) dispatch over the aabb prims (the n-body tiling: each workgroup stages 64 prims into workgroup memory per round — a naive per-lane scan of global prims is latency-bound, measured WORSE than the BVH it replaces); above it, the LBVH build + descent. Both interpolate the same candidate/emit WGSL (ownership, nearest-K + static pin, ascending sort, INVALID fill), so the per-eid pair blocks are identical and warmstart carries across a regime flip; both are exact at any N, so the stale switch is correctness-safe in both directions. The small kernel's query box is its own prim shrunk by `SPECULATIVE_DISTANCE` (the skin is prim-only, recovering the descent's tight query exactly). **A runtime threshold crossing is gated, both directions:** the gym pile `regime-cross` check (driven by the bench's `regime-cross` scenario row — thresholds set at the pre-spawn live count, so the spawn-despawn flow crosses smallN AND ldsN up and back down, with the originals' poses held across both flips). Its witness is the `PhysicsStep.regimes` getter (set in `record()`), NOT spans — the profiler holds a non-firing pass's last span, so a span can't see the flip back.
- **The CSR + coloring tail follows the regime (C1.1).** The small regime runs the whole tail — count-clear, CSR count→scan→scatter, and the incremental greedy — as ONE single-WG dispatch (`CSR_COLOR_SMALL_WGSL`): each phase is the multi-WG pass's logic verbatim strided over 256 lanes under in-kernel barriers, the count clear touches live eids only (dead eids' counts are never read), the scan chunks over the LIVE count (the standalone scan chunks over `maxBodies`, idling most lanes at small N), and the coloring inverts staging (priors read from `colors`, chosen staged in `colorScratch`, committed after a barrier — a single WG doesn't need the multi-WG pass's 256 KB snapshot copy, which exists only because its workgroups race). The joint hard-conflict repair keeps its own snapshot+pass rounds in both regimes; the fused pass reports under the `phys:csr` span (`phys:coloring` reads 0 in the small regime). 10 storage bindings — AT the floor like the primal, so a new constraint type must reuse the merged adjacency, never add a binding here. Measured (Lovelace, min-over-reps): the tail 89 → 29 µs at 256 bodies but 84 → 94 at 1000 — one workgroup is one SM, so the fused tail grows ~linearly and **crosses over the multi-WG passes at ~850 bodies**; the regime's front-end win still dominates, so it ships. The top-of-regime gap is PRICED on Lovelace (C1.4 A/B at 1000 bodies, `--audit --smallN 0`): ~6 µs against a ~56 µs net regime win (step 411 vs 467), so the lever (multi-WG count/scatter + a dynamic-chunk scan + clears folded into earlier passes — never a third threshold) stays unbuilt there; the M4 A/B was skipped (C1.4 wrap-up) — gameplay scenes sit far below the crossover, so reprice only if a Metal scene measures over budget at the regime top.
- **The solve has an LDS-resident regime too (C1.2), defaulting to the tiny end.** At a frame-stale live count ≤ `StepParams.ldsN` (default `LDS_N` = 64; 0 = always looped; clamped to `LDS_CAP` = 512, the 16 KB workgroup-memory capacity at 28 B/body), `record()` replaces the whole iters × colors primal/commit/dual (+ joint dual) block with ONE single-WG dispatch (`SOLVE_LDS_WGSL`): poses live in workgroup memory across the loop (pos as 3 split f32 columns + quat), the eid→dense map rides the unused `solveOut` buffer rebound as `array<u32>` (the kernel is exactly AT the 10-storage-binding floor — a new constraint type must not add a binding here either), each color commits register-staged poses after a barrier (the double-buffer), and the used-color count is an in-kernel `atomicMax` (fresher than the looped path's readback, no color UBO). The solve math is shared chunks (`solvePose` / `dualSlot` / `jointDualOne`), so GPU == oracle holds on both paths; the block reports under `phys:primal` (`phys:dual`/`phys:joint` read 0 in this regime, the `phys:csr` precedent). **Why the default sits at 64, not the 512 capacity:** measured on Lovelace (min-over-reps, iters=4), the kernel is parity at ≤~64 bodies and loses ~linearly above (solve block 0.157 vs 0.132 ms at 130 bodies, 0.426 vs 0.148 at 502; the looped block is ~0.13–0.15 ms flat at any count ≤1k) — pose residency removes only the LAST link of a color phase's dependent chain; the record/CSR storage latency remains, serialized on one SM, and the in-kernel dual forfeits multi-WG parallelism. The C1.4 M4 sweep (`--ldsN 0/512`) refuted the hoped-for Metal boundary recovery: the LDS block measures 82/189/349 µs at 66/258/502 bodies vs looped 129/134/170 — crossover ≈ 100–150, the same storage-latency law despite the 4× boundary constant — so 64 is the default on every device (a one-number `ldsN` re-sweep is the lever if a future device differs). Sweep gotcha: the audit's count=64 cell is 66 bodies > the gate, so a default-config audit exercises the looped path only. A live body past `LDS_CAP` mid-staleness falls back to its storage pose and skips its solve one frame (the BODY_MARGIN class).
- **Quantization defers; data layout doesn't.** The body + contact buffers are the **SoA cols-buffer** (`bodies[col*capacity + eid]`, `pairContacts[col*recordCap + rec]`, gpu.md consolidation #1) — the rebuild contract's *no AoS body buffer / commit to SoA*, a foundation. What *does* defer: **quantization** (f32-first; packed formats only when a pass benches ≥5× off the bandwidth floor, gpu.md rule 8 — a storage-boundary format choice, not a reshape). The CSR-fed **subgroup-per-body** solve (VBD §3.10) is the one algorithm still deferred — but per-body CSR adjacency (B1) moved the primal out of the compute-bound regime the subgroup shape targets, so it's gym-only large-pile insurance, gated on a measurement showing it beats the thread-per-body CSR solve at high valence (not below the microbench's ~valence-12 crossover).

## Layered augmented Lagrangian, one solver

The `Force` law (`updatePrimal`/`updateDual`) is shared; each layer adds terms to the dual update, the primal is never rewritten. A new constraint type (joint/spring/character) is a new `Force` / a new appended contact tag, not a solver change. **The binding floor (`maxStorageBuffersPerShaderStage = 10`) is the hard constraint on adding a constraint type.** Springs + joints therefore SHARE one authored-constraint adjacency: `constraintCsr` + `constraintList` (kind-tagged entries, `vec4[2].y` = `KIND_SPRING`/`KIND_JOINT`, the primal/coloring branch on it). The primal is **exactly at the floor** (10 storage buffers — it adds the one `jointRecords` binding a joint's persistent λ/penalty/c0 needs); the **coloring is AT the floor and adds nothing** (it reads the merged adjacency for kind-agnostic avoidance only — the joint hard-conflict repair is a SEPARATE pass). So the next constraint type (character contacts, voxel) must REUSE the merged `constraintCsr`/`constraintList` or merge into a record buffer — never add a primal/coloring binding, or the pipeline fails to create on the floor. **Joints are dispatched one-thread-per-joint** (init + dual), so per-joint state that both endpoints' primal reads (λ/penalty/c0/torqueArm/active) lives in `jointRecords` (AoS), not the per-endpoint entry; `torqueArm` + the recycle `active` flag are GPU-written, ∞ stiffness is the `1e30` sentinel (`> 1e29` = rigid). The recycle-version guard + the once-only anchor-coincidence guard (the GPU analog of the oracle `joint()` throw — deactivate `active=0` + bump `counters[2]`, since the GPU can't throw) both live in jointInit. **jointInit dispatches per-joint, NOT gated on the live count, so it MUST gate on the per-eid `seeded` flag** — else the first fixed step (which at 60 Hz runs before the first pack seeds) reads a zero-init `bodies` slot (`half = 0` → reach ≈ margin) and the anchor guard wrongly rejects every valid joint. **WGSL keyword trap: `active` is reserved** (use `act`); `bun bench` compiles WGSL on the real device (Tint), so a bench run is a genuine shader-compile + pipeline-creation gate, not only a visual one.

**The GPU runs warmstart-only.** The penalty and dual-only layers live only in the **CPU oracle** (`tests/avbd/solver.ts`, `makeSolver({layer})`), where the layered oracle tests (`oracle.oracle.ts` resting-`mg/k`, dual-rest, the cold-vs-warmstart convergence proof) exercise all three; the GPU `step.ts` is the warmstart layer only (no `StepParams.layer`, no `warmstart` uniform branch). One layer-coupled constant: the `dual` oracle layer cold-starts `λ` each frame so it needs `betaLin = 1e5`; the shipping `warmstart` layer uses the canonical `1e4` (persistence does the convergence work).

**Colored GS, not Jacobi** (paper + community + a 1D probe all agree). Same-color bodies share no constraint, so solving them in parallel from stale neighbors *is* sequential GS — deferring *all* updates (global Jacobi) discards that ordering and destabilizes. Coloring isn't only for write-safety; it preserves the sequential solver's stability. Chebyshev is out (overshoots on active contacts). The solve commits **double-buffered-within-color** (the grounded method: paper Algorithm 1 lines 22–24, webphysics `commitBodySolveKernel`, the oracle's `primalColored` all do this): the primal solves a color into the `solveOut` scratch reading the committed `bodies`, then a commit dispatch applies `solveOut` → `bodies` for that color. A conflict-free color is unaffected (no same-color pair shares a constraint); a folded body past the cap — where a greedy sweep can't guarantee a conflict-free coloring — is now a *clean* Jacobi (both bodies read the color-start pose) instead of write-in-place's same-color read-write data race (UB in the WGSL memory model). The cost is a commit dispatch per color, roughly doubling the primal-related dispatch count; reference fidelity + removing the UB outweigh the saved dispatch. (An A/B on nvidia/lovelace found write-in-place *benign* there — no cross-invocation write visibility within one dispatch, so it resolved as the same deterministic Jacobi — so this is a soundness/portability fix across the AMD/Apple floor, not an observable nvidia fix.)

## Joint guards + the grab

Two construction-time joint guards reject an energy-injecting authoring — the oracle `joint()` throws, the GPU `jointInit` deactivates + bumps a counter (the GPU can't throw). Both mirror the contact all-static dual guard's posture: a constraint no dynamic body can resolve must not ramp.

- **Both endpoints non-dynamic (`mass ≤ 0`, the world anchor counting as static) → reject (`counters[1]`).** A joint between two static/kinematic bodies is never satisfiable by the primal (both skip it), so the dual ramps its penalty + λ unbounded; if such an endpoint is later released (made dynamic) the accumulated λ yanks it. The GPU re-checks **every frame** (a both-static joint is permanently unsatisfiable, gated behind the seed gate so an unseeded body's zero-init mass can't false-trigger it) — so `counters[1]` is a **persistent gauge**, not a one-frame blip a per-frame-cleared counter + lagged Mirror readback would miss. The anchor-coincidence guard (`counters[2]`, the rope-explosion) stays act==2-fresh-only (it's pose-dependent). Gated: `oracle.oracle.ts` "both-static … fails loudly", the gym `constraints` both-static gate (real GPU, reads `counters[1]`).

### The grab dangles from a WORLD anchor — no anchor body, no contact (avbd-demo3d's design)

The grab pins the held box to a **world-space anchor point** (`a = WORLD`, `rA` a world point — `joint.cpp`'s `bodyA == null`) by a **soft spherical** joint, NOT a kinematic anchor body. The box **dangles** (rotation free), the reference's sandbox feel; and with no anchor body there is **no anchor↔box contact**. Holding the cursor still does no work, so the held pendulum can only DISSIPATE — it never injects energy or flails chaotically.

- **The original grab flail had two causes, both removed by the world anchor.** The old grab used a kinematic-anchor *body* (a sphere embedded in the box surface) → a **contact** that shoves the box (the dominant flail), plus the spherical joint's free rotation. The world anchor (no body) deletes the contact; the remaining spherical dangle is a *damped* pendulum (BDF1 dissipates it), which is the intended feel, not a bug. Measured (oracle): a world-anchor spherical grab whipped hard then held damps (peak ω falls window-over-window); a kinematic-anchor *body* stays more agitated (its contact keeps feeding energy).
- **Implementation.** A `JointDef` with `a = WORLD` (any `a < 0`) makes `rA` a world point; `setJointAnchor(index, x, y, z)` moves it each frame (the reference's `drag->rA = …`), writing ONLY the record's `rA` lane so the warmstart + active flag survive — NOT a re-author (a re-author would re-run the anchor-coincidence guard mid-drag and wrongly reject a moved anchor). On the GPU the world is the sentinel `WORLD_ANCHOR` (`0xffffffff`): it gets **no constraint-list entry** (only `b` does, pointing at the sentinel), no version, no seed gate, identity orientation, size/mass/radius 0. **`jointContrib` reads `rA`/`rB` from the per-joint RECORD, not the list entry** — the record is the single source of truth, so a moving anchor is seen by the primal (a static joint's record == its list entry, so non-world joints are unchanged). Oracle: `a: Body | null` in `joint.ts` (null = world); the solver skips the null endpoint's adjacency.
- **Hold the anchor at the hit point until ONE fixed tick has run.** The grab joint's one-shot construction guard judges `|anchor − pin|` at its init step (the next fixed tick after `setJoints`), and a thin link's reach is centimetres — dragging/reeling the anchor in the render frames before init moves it past the reach and the grab is rejected silently (a dead grab: nothing happens). Gate the first anchor move on `state.time.fixedTick` advancing past the grab tick (the sandbox gun's `grabTick`).
- **The pick is occlusion-aware (`grabHit`, `physics/core`).** The cast keeps statics (they're the occluders — don't re-filter them out of `bodyCandidates`) and returns the nearest hit only if dynamic, so a wall blocks the grab. The caster's own `mass ≤ 0` capsule MUST be `exclude`d or it self-occludes the crosshair ray (gun + `pickDrag` pass `exclude`).
- **Gated.** `oracle.oracle.ts` "the grab dangles from a world anchor without injecting energy; a kinematic-anchor BODY flails" (dangles + DAMPS + bounded + finite; the body-anchor stays more agitated); the gym `constraints` re-author + sphere-grab checks (kept slots, radius-aware reach) gate the grab on the real GPU; the sandbox showcase's gravity gun (`gun.ts`) is the live real-GPU exercise — world-anchor lift, fast-drag stable dangle, and the occlusion-aware pick (a ray up through the static floor returns null, not the box behind it).

## Box settle — no edge-wedge (investigated, no bug found)

The reported "a box balances tilted on an edge (~30°) instead of toppling" does **NOT reproduce** in the f64 oracle OR on the real GPU. Verified exhaustively: single boxes at every release tilt 20–70° (rock-back + topple-to-face), vertex-down drops (the hardest narrowphase), ramps, and ledge-overhangs with the COM past the support edge — all settle flat (or correctly fall off) at iters 4/6/10 and friction 0.5–0.9. GPU == oracle per-step holds (the gym single-step gates), verified at the **shipped iters=6** too (the ship value, where the corpus gates' iters=10 wouldn't have caught a divergence). **The likely report source is a visual misread:** a box resting **flush** on a 25° ramp is correctly tilted 25° (its bottom face on the ramp, COM over the contact face) — it *looks* tilted but is on a face, not wedged on an edge, the correct "tilted box" distinct from a flat-ground wedge. **Don't re-hunt this as a solver bug**; if a genuine wedge ever surfaces, it'd be a chaotic multi-body f32 contact-order effect (physics.md "Two more f32 realities"), not the single-box over-stabilization the report imagined.

## Dispatch count is the binding cost, not the solve span

Colored GS is `iterations × dispatched-colors` dispatches, each a CPU-encoded compute pass — and the per-dispatch **encode** cost, not the GPU solve span, is the binding axis (cost model + measured figure in gpu.md "Dispatch count is a first-class cost": ~5.9 µs all-in/dispatch on desktop D3D12, multiples higher on the Deck). Against a ~1 ms physics budget on the 4090 — sized so the Steam Deck still hits reasonable rates — a hard solve's ~90 dispatches ≈ 0.5 ms is half the budget before the GPU solve. So the perf gate is frame/encode cost + a dispatch-count ceiling, not the `phys:solve` timestamp; the phase-0 ns/body/iter microbench is a single flat dispatch (compute-bound regime only) and is silent on this. Discipline that follows:

- **No per-color `advanceColor` dispatch.** The current color rides a dynamic uniform-buffer offset (`colorUbo`); the color value never costs a dispatch.
- **One compute pass per iteration, each color a primal+commit pair (double-buffer).** `record()` opens one pass per iteration and loops the colors inside; per color the primal solves into the `solveOut` scratch (reading committed `bodies`), then the commit applies `solveOut` → `bodies` for that color so the next color's primal sees it. Consecutive same-pass dispatches' write→read hazards (primal→commit on `solveOut`, commit→next-primal on `bodies`) are barrier-ordered — the single-step gate confirms the cross-color commit matches the oracle. Dispatch count per iteration is `dispatchedColors × 2` (the commit doubles it); a pass *per color* is the thing to avoid. Why double-buffer over write-in-place (a soundness/portability fix across the AMD/Apple floor, benign on nvidia/lovelace, the CPU oracle gating colored-vs-sequential equivalence): the "Colored GS" paragraph above.
- **The dispatched colors are the readback-bounded count, not the static cap (the readback-bounded color loop).** The coloring may use up to 32 colors but folds bodies past `maxColors` (`bid % maxColors`); the primal loops `min(maxColors, usedColors + COLOR_MARGIN)` color-passes (`PhysicsStep.boundColors`), where `usedColors` = the greedy's max dynamic color + 1, published to `colorCount` by `colorize` (one `atomicMax`) and read back **frame-stale** through a Mirror (the plugin's `colorMirror` → `boundColors` each fixed tick). A sparse scene (greedy ≤ 2 colors) dispatches ~2-3 of 8, the saving on the overhead-bound common case; a dense pile (chromatic 9-11) caps at `maxColors` unchanged. `usedColors ≤ 0` (no readback yet) keeps the full cap — the safe cold-start. It's a **loop resize, never a solve-math change** (GPU == oracle holds); the standalone gates drive `record` without a readback so they stay at full dispatch. **The coloring bootstrap must cold-seed (`0xffffffff`), not reuse the prior frame's eid-colors** — a warm/identity seed keeps each conflict-free body's scattered color so `usedColors` never collapses (≈8 even when sparse), killing the saving (correctness-neutral, a pure dispatch-count regression). `COLOR_MARGIN` (= 1, gym-gated) covers the readback's 1-2 frame staleness — a frame that densifies further under-dispatches once, a soft convergence dip the next readback catches (a frame-staleness margin, the profiler-counter class, *not* a tuned solver tolerance). Reported by the bench (`scripts/physics-bench.ts` reads `dispatchedColors` from the gym `measured` payload); GPU == oracle holds at any dispatched count (the standalone gates run full dispatch).
- **Body passes dispatch indirect off the live count; the primal/commit color loop dispatches DIRECT off its frame-stale readback.** `clampCount` writes `dispatchArgs = [ceil(count/64), 1, 1]`; the once-per-step body passes (aabb / broadphase / inertial / coloring / velocity / compose) `dispatchWorkgroupsIndirect` off it — exactly the live count's workgroups, never the `maxBodies` pool — and the collide / dual / CSR passes dispatch indirect off `pairArgs`. The **color loop** is different: it's `iters × colors × 2` dispatches (the dominant block), and an indirect dispatch costs ≈ 2× a direct one (Dawn's injected validation pass, gpu.md "Indirect ≈ 2× direct"), so `clampCount` also publishes the clamped live count to `colorCount[1]` — riding the same Mirror readback as `usedColors` — and the primal/commit pair dispatches `ceil((liveCount + BODY_MARGIN) / 64)` workgroups **direct** (`PhysicsStep.boundBodies`). This is safe without a path-select: over-dispatch is correctness-free (every body pass early-outs on `d >= eids[0]`, so cold start = full-cap direct, the `boundColors` pattern — no GPU-count-contract fight); only under-dispatch needs the margin — a spawn burst past `BODY_MARGIN` (64, one workgroup) inside the readback's 1-2 frame staleness skips the burst body's primal for one frame, which the next readback catches (the gym `pile` spawn-despawn gate covers it). The standalone gates drive `record` without a readback and stay at full-cap dispatch. The early-out-direct shape is webphysics's `dispatchBodyCount` (`reference/webphysics/.../avbdState.ts`); ours sources the count from the readback instead of a CPU-managed count.
- **The O(count·contacts) scan is gone — CSR, not subgroup, was the fix.** Per-body CSR adjacency makes the primal read only a body's own contacts (O(valence)). That moved `phys:primal` out of the compute-bound regime, so the subgroup-per-body solve (which parallelizes per-body compute) is now measured-marginal at typical valence — gym-only insurance, not a default.
- **Single-WG megakernel (ladder rung 1) REFUTED on desktop — built, measured, deleted 2026-06-10.** Fusing the whole iters × colors primal/commit + dual + joint-dual block into ONE single-workgroup dispatch (Bullet 3 `BatchSolveKernelContact` shape, `storageBarrier()` between color phases, strided over bodies, shared solve math so the full gym gate set ran green on both paths with identical GPU == oracle errors) measured **parity-within-noise at iters=4 and slower at iters=10** on lovelace (min-over-3, gym pile pyramid 64/128). The diagnosis behind the rung was wrong: the color loop's ~9–11 µs/dispatch is not removable launch/encode overhead — it is each phase's **dependent storage round trip** (CSR → contact records → both bodies' poses, serialized behind the phase boundary because the next color reads what this one wrote), and that chain costs the same at a dispatch boundary as at an in-kernel barrier. The small-N solve is **latency-bound on a serial phase chain**, not dispatch-, compute-, or BW-bound. Consequences: dispatch-count work pays only when it deletes *phases* (rung 0's readback-bounded color loop; shipping iters=6), never when it repackages them; don't rebuild any megakernel without the lever that removes the round trip itself. **The LDS-resident form WAS built (C1.2, 2026-06-10 — the "Storage" bullet above) and sharpened the diagnosis:** poses-on-chip removes only the last link of the chain — the record/CSR storage latency remains, serialized on one SM — so even the LDS form is parity-at-64/regression-above on Lovelace and ships defaulted to the tiny end (`LDS_N` = 64). The C1.4 Apple cell ran that sweep and confirmed the law on Metal (crossover ≈ 100–150 despite the 4× boundary constant — storage latency, not the boundary, is the cost); the Deck is the one unswept floor, and the sweep stays a one-number `ldsN` rerun.
