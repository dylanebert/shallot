# Tumble backend

`standard/tumble` is shallot's **built-in physics**: the engine itself at `standard/tumble/engine/` (a from-scratch TypeScript port of Erin Catto's Box3D, hot path in the Rust wasm kernel at `rust/tumble/`) plus the glue at the module root that puts it behind the `standard/physics` substrate (`physics.md`) as a `PhysicsBackend`. It is the **default** backend: gameplay-complete, runs in wasm on the CPU (no GPU device to step, so it's testable headless), meets the WebKit floor, lands ~2× native-C single-thread, and multithreads wherever the host affords shared memory. `AvbdPlugin` is the massive-scale GPU swap-in (`avbd.md`); `TumblePlugin` + `Tumble` ride the main barrel (`exports.md`). `Tumble.world` (the live wasm `World`) is the imperative escape hatch for anything past the atomic `PhysicsBackend` core; `Tumble.body(eid)` returns the marshaled body handle.

The glue sections come first; "The engine layer" below is the engine's standing how-to (bit-exact contract, kernel boundary, MT, fixtures, upstream pin).

## The wasm world is a singleton — dispose is load-bearing

There is **one live `World`** per process, held at module scope in `standard/tumble/index.ts` (`Tumble.world`, exclusive). `TumblePlugin.warm()` constructs it; **`dispose()` must free it** — the wasm kernel does not garbage-collect, and a hot reload / scene swap that constructs a second `World` without disposing the first leaks the wasm heap and double-steps. Every method reads the module-level `world` / `bodies` fresh (never captures it), so `Tumble.body` stays correct across a warm rebuild. The reload-conformance roster (`lifecycle.test.ts`) pins this.

**Creation order is load-bearing for determinism.** The engine's per-step world-state hash depends on body creation order (the colored solver's contact ordering keys on it), so the marshaling must create bodies in a stable eid-sorted order. A ragdoll's `buildHuman` recipe fixes its capsule + joint creation order for the same reason.

**The interpolated compose freezes a hair short of a body's final pose at sleep** — the `ComposeSystem` renders `lerp(prev, curr, fixedAlpha)`, and a body's *last* firehose write is the frame it stopped moving, at `fixedAlpha < 1`, so the slot settles `(1−α)·(last tick's delta)` short of `curr`. Bounded well below the sleep threshold, cosmetically invisible — don't chase it as a jitter bug.

**Eid identity across a same-update destroy + create is stamp-checked, not presence-checked.** `SyncSystem` stores `state.stamp(eid)` beside each marshaled handle; a stamp mismatch on a still-member eid means the slot was recycled to a NEW body — the stale handle is destroyed, the entity re-marshaled, and the derived kinematic state (`kinPrev`/`movedThisTick`) evicted. Presence diffing alone couldn't see the realias (membership never drops), which used to leave the old tumble body simulating entirely. The stamp mechanism is `ecs.md` "An eid is a borrow"; the gym `backend` recycle station gates it on real GPU under both backends.

## Constraint mapping — settle-to-equilibrium is the swap-parity behavior

The substrate hands `SpringDef` / `JointDef` (backend-neutral) to `syncSprings` / `syncJoints` (`joints.ts`); this module maps each to a tumble constraint. The mappings are chosen so a scene **behaves the same after a backend swap** — same rest pose, same settle — not bit-for-bit (cross-backend parity is behavioral, never bit-exact):

- **`Spring` → DistanceJoint-with-spring.** Stiffness (N/m) converts to tumble's soft-constraint frequency: `hertz = √(k / m_eff) / 2π`, `m_eff = mA·mB/(mA+mB)` the pair's reduced mass (a non-dynamic endpoint contributes ∞ mass, so `m_eff` = the dynamic side's mass), reproducing the N/m force law's static extension under load. The joint is **critically damped (`dampingRatio: 1`), NOT the literal undamped elastic law** — AVBD's BDF1 integration heavily damps its own spring, so settle-to-equilibrium (not a ringing oscillation) is the behavior that matches across the swap.
- **`Joint` → Spherical / Weld.** `stiffness-ang: 0` (the default) → `createSphericalJoint`; rigid past the `RIGID_THRESHOLD = 1e29` ∞ sentinel → `createWeldJoint` (an intermediate angular stiffness is a documented hertz-based approximation via the same reduced-mass conversion, unit-arm `I_eff ≈ m_eff`). The **spawn-relative rotation is captured** at construction so the weld holds the authored relative pose.
- **Content-keyed diff-preserve.** `syncJoints` diffs the authored set against the live tumble joints by def **content**: an unchanged constraint keeps its live (warm-started) joint; only changed/new defs rebuild. This is the substrate's re-upload-on-signature-change (`physics.md`) carried down to the tumble joint objects — a re-author under load doesn't drop a loaded chain's warm state.
- **Both backends reject an unsatisfiable constraint** (one no dynamic body can resolve — both endpoints non-dynamic, or a non-`Body` ref). AVBD deactivates + counts it (its GPU can't throw); **this path `console.warn`s + skips** (`hertz === 0` / no dynamic endpoint / non-Body ref). Point the warning at the diagnostic, never the brand.

## The marshaling gate — dual-run hash equality

The integration contract between the ECS and the engine is **per-step world-state hash equality**: the same scene run twice — once through `TumblePlugin` (marshaling `Body`/`Spring`/`Joint` entities), once by hand through the raw engine `World`/`Body` API with identical literal values — stepped in lockstep, must produce the same `hashWorldState` at every step (`tumble.test.ts`, run each build to completion recording hashes, then compare). The reference arm shares **no code** with `marshal.ts` / `joints.ts`, so any pose/mass/order bug in the plugin's marshaling diverges the hash. This is the tier-`bun test` marshaling gate (headless, no GPU); it covers one of each Spring/Joint mapping (a distance spring, a pendulum bob, a welded arm). The engine exports `hashWorldState` for exactly this.

## Ragdoll — hybrid authoring (substrate bodies, escape-hatch joints)

A ragdoll is the worked example of the substrate-plus-escape-hatch split, and its recipe stays **example code** (the gym `ragdoll` mode + the ragdoll recipe in `examples/recipes/`), never an engine module — a hardcoded humanoid isn't engine API, and examples can't cross-import, so the compact recipe is duplicated across the harness + teaching tiers. The engine surface it needs is only two seams: `Tumble.body(eid)` (the escape-hatch handle lookup, `bodies.get(eid) ?? null`) and `skinMatrix(pos, quat, invBind, out?)` on `gltf/core` (the object-space skin compose a pose producer feeds `LiveSkin.writePalette`). A general auto-ragdoll-from-skin builder is genuinely-novel unscoped work; don't infer it from this specimen.

- **Bones are substrate `Body` capsules** (so writeback, `pick`, and the character sweep see the ragdoll); **joints ride `Tumble.world`** (`createSphericalJoint` with cone/twist limits + motor, `createRevoluteJoint`, `createFilterJoint` — deliberately richer than the substrate `Spring`/`Joint` mapping, which does not grow to match). Joints wire **one-shot post-marshal** via `Tumble.body`, with analytic spawn-derived local frames (a live `getLocalPoint` would fold the pre-wire free-fall ticks into the anchors; tumble's cone/hinge axis = joint-frame Z).
- **Bodies a deferred joint connects must spawn non-overlapping.** The fixed tick(s) between body creation and the post-marshal wire mint a persistent contact between the pair, and that contact fights the joint from then on. Box3D purges contacts between connected bodies only in the runtime `SetCollideConnected` setter, never at joint creation, so the fix is authoring geometry: spawn the bodies apart (anchor a driven body to ground it does not overlap). Two concentric bodies wired a step late stall against their own contact regardless of `collideConnected`.
- **The pose math is a rigid bone-delta, and the glTF IBM cancels at runtime.** With each joint rigidly following its ragdoll bone, `palette_j = boneObjNow_{b(j)} · boneObjBind_{b(j)}⁻¹` — the joint's inverse-bind cancels against its bind-global, so the producer needs only each **bone's** object-space bind (captured at build) + the bone→joints map, never per-joint IBMs at runtime. So `skinMatrix(relNowPos, relNowQuat, invBind)` = `compose(relNow)·invBind` where `invBind` is the **bone's** root-relative bind inverse. Identity at bind (`relNow == bind → I`) is the headless test's anchor. Root-relative to the root bone (pelvis) keeps the palette bounded so the live-skin reach-sphere cull holds as the ragdoll falls; the instance `Transform` tracks the pelvis (`pelvisNow · pelvisObjBind⁻¹`).
- The pose producer reads `readBody` per bone (fresh under tumble), keeps its own prev/curr pair, nlerps at `state.time.fixedAlpha` (`nlerpShortest` from `compose.ts`), and writes the palette each fixed tick.

## The engine layer

`engine/` shipped standalone as the `tumble.js` npm package before being folded into shallot (2026-07); comments calling it "the box3d TS port" or "tumble.js" are provenance, not a dependency — the engine is self-contained source with committed base64 wasm, no runtime deps. All 9 joint types run in-kernel under MT; serial TS joints remain only on the no-pool single-thread path. Everything below is the engine layer's how-to.

## The contract: bit-exact f32 parity

Regression is **per-step world-state hash equality** against the Box3D C reference, not tolerance bands. The reference build is the box3d checkout on branch `harness`, configured `BOX3D_DISABLE_SIMD=ON` (default overflow **off**) — the colored solver: graph coloring + the wide 4-lane convex path + serial mesh/overflow spill, which the port mirrors per-lane. `DISABLE_SIMD`'s scalar `FloatW` is bit-identical per lane to the SIMD build, so the fixtures pin the shipping wide-simd wasm path too. The `graph.ts` `FORCE_OVERFLOW` const (and the C `BOX3D_FORCE_OVERFLOW` option) still route everything through the serial scalar overflow color — the earlier parity config, kept as a fallback knob, not the default. The engine replays each scene and asserts its FNV-1a hash equals the fixture's, per step, including sleep-step indices. (The reference checkout is the mint instrument, present only in the engine author's workspace layout — the committed fixtures carry the contract without it; see Fixtures.)

This only holds if the port's f32 arithmetic is bit-identical to the C build. The disciplines that make it true:

**1. fround every f32 op.** JS numbers are f64; f32 is emulated by `Math.fround`. Alias it once per module (`const f32 = Math.fround`) and wrap **every** arithmetic operation, one op per wrap, mirroring the C expression tree exactly. Box3D builds with `-ffp-contract=off` and JS has no FMA, so there is never a fused multiply-add: `float t = a*b + c*d` ports to `const t = f32(f32(a * b) + f32(c * d))`, three wraps. Storage is `Float32Array` SoA (rounds on write), so values read back from a column are already f32 — but never lean on that mid-expression; wrap each intermediate. The double-rounding theorem guarantees `f32(a op b)` equals native f32 `op` for `+ − × ÷ √` when `a`,`b` are f32-valued. `Math.sqrt` composes the same way (`f32(Math.sqrt(x))`); transcendentals are ported op-for-op from Box3D's own portable trig (`b3Atan2` minimax, `b3ComputeCosSin` Bhāskara in `math_functions.c`), never JS `Math.sin`/`atan2`.

**1a. fround every non-exact literal too.** A float literal that isn't exactly representable in f32 is an f64 operand until you round it, and it multiplies/adds differently than the C's `f`-suffixed constant. `0.4f` in C is `fround(0.4)` = `0.40000000596…`; JS `0.4` is the f64 value, and `f32(0.4 * mass)` ≠ `f32(f32(0.4) * mass)` for some masses. Round the literal first: `f32(f32(0.4) * mass)`. This bites hardest when the literal **compounds** — `r = f32(f32(0.95) * r)` in a taper loop drifts a full ULP per iteration and eventually crosses a rounding boundary, changing a hull vertex and every inertia derived from it. Exactly-representable literals (`0.5`, `2.0`, `0.25`, any `k/2ⁿ`) are already f32 and need no wrap; `0.4`, `0.95`, `0.1`, `0.3`, `1/3` do. When in doubt, wrap — `f32(x)` on an already-exact value is a no-op. Three separate bit-exact divergences (sphere-inertia `0.4`, trees-taper `0.95`, geometry rounding) traced to a bare non-exact literal; treat every literal that reaches f32 arithmetic as suspect.

**2. Mirror the C scalar branches, not the SIMD intrinsics.** The port targets the `DISABLE_SIMD` build, whose min/max are explicit comparisons: `b3MinFloat(a,b)` is `a < b ? a : b`, `b3MaxFloat` is `a > b ? a : b`. Port those ternaries verbatim. **Do not use `Math.min`/`Math.max`** — they diverge on NaN (`b3MinFloat(NaN, b)` returns `b`; `Math.min(NaN, b)` returns NaN) and on signed zero (`b3MinFloat(-0, +0)` returns `+0`; `Math.min(-0, +0)` returns `-0`). Any place the C scalar path branches on a comparison, the port branches the same way.

**3. fround the API boundary.** User-facing defs/configs arrive as f64; round every float field once where it crosses into the engine (`froundConfig` at create, the same rounding in runtime setters) so internal arithmetic always starts from f32 values. Several bit-exact divergences traced to an unrounded def field reaching the solver.

**One documented trig deviation:** `createWaveMesh` mirrors the one upstream geometry helper that calls libm `sinf` (not the portable trig), so the port uses `f32(Math.sin(x))`. The observed `Math.fround(Math.sin(x)) === sinf(x)` equality is a platform coincidence, never a contract — wave geometry is tested structurally, never bit-exact, and a fixture that needs libm-derived geometry gets it emitted from the C generator instead (the trees `emitAux` ground-vertices pattern).

## The wasm kernel

The hot path is a hand-written Rust crate at `rust/tumble/`, compiled `wasm32-unknown-unknown` and base64-inlined into `engine/kernel.wasm.ts` (pure JS, no separate asset) via `bun run scripts/build-tumble-kernel.ts` (from `packages/shallot`). `init()` async-compiles it once (`engine/kernel.ts`); `step()` is synchronous after. The crate also builds natively, so each ported phase is `cargo test`-verified against gold vectors extracted from the C reference (a `*_gold` target in the reference checkout → committed `engine/<name>.gold.json`, regenerated by `bun run gen-tumble-gold <name>`), then the wasm path is fixture-gated end to end.

**Bit-exactness in Rust.** Rust f32 is native IEEE — no `fround` discipline (never call `mul_add`; Rust never contracts otherwise). But two C traps still port verbatim: `f32::min`/`max` diverge from `b3MinFloat`/`b3MaxFloat` on `NaN`/`−0` — port the ternaries, don't use them; and there is no libm — port box3d's portable trig. Mirror the C **file/function structure** in Rust so an upstream solver delta hand-ports as a readable diff.

**The kernel/TS boundary rule.** Per-entity hot loops over columnar data go in-kernel; graph/world mutation, islands, contact events, and the public API stay TS. The FFI is **phase-granular** — ~15 crossings/step, proven a non-cost (batching ~400→~15 moved the bench zero). A per-contact crossing is the one forbidden shape.

**AoS-record columns + the gather lesson.** Body state is one AoS record per body (power-of-two f32 stride, 16 B-aligned), gathered *randomly* through contact indices — a contiguous record is 1–2 cache lines per touch. The wide solver gathers 4 records and transposes in registers. **Gather record-wise, never field-wise:** per-lane scalar loads + inserts lose to the overflow-scalar path in JS-hosted wasm (simd128 has no gather instruction); 4 aligned `v128` record loads + an unpack-only 4×4 shuffle transpose recover ~1.9× (JSC) / 2.5× (V8), bit-identical (`contact_wide.rs`). Read through **raw pointers** (bounds-check elimination is part of the win). Null/static lanes remap branchlessly to a **trailing identity record** the body region reserves past `bodyCap`. Hardware gather is a red herring — AVX2 `vgatherdps` is no faster than scalar loads even natively.

**Persistent regions + relocation.** Resident columns live in low linear memory, ordered body → fatAABB → shape → manifold → geo, each grow-only. A lower region's grow relocates everything above it: one overlapping `core::ptr::copy` of the whole span + rebasing the kernel statics (contents are element-relative, so nothing inside the span is fixed up). **A growing region must also preserve its *own* columns** — only the base-anchored column survives a stride re-space for free; the rest need a top-down copy (a dropped-FLAGS bug free-fell whole scenes once capacity passed 16). Anchor each region from the one below via `persistent_base()`/`region_top()`. The shape column (`shapes.rs`/`shapecolumns.ts`) is written at shape create/destroy only, and it is never zeroed — index it via the awake head-shape lane → `next` chain, never a `0..cap` sweep. **The port has no geometry setters** (box3d's `b3Shape_SetSphere`/`SetCapsule` were never ported); porting one later means writing the shape column at that site — same for a future `b3Shape_SetFilter` and the tree's category halves.

**View-refresh (the read-side dual).** TS views over resident columns are the `ManifoldPointView` store-indirection pattern (read through the store every access), **not** `ColumnState`'s captured-array pattern. Refresh keys off the **grow-capable call** (`reserve*`), never off detachment — a body-grow relocation without a page grow leaves views *attached but stale* (silent wrong reads). Additionally, any resident-column read **inside a loop that can `memory.grow`** (the mid-collide manifold `alloc`) must re-derive the views on detachment before the next read — the plain-object sims this replaced were immune, column-backed views are not (a NaN-transform bug found this). Guard on detachment (`length === 0`), not per-iteration, or you reintroduce churn. Third leg: a reserve-keyed refresh at step top is insufficient for a read pass that runs *after* a mid-step grow (collide's manifold alloc, solve's arena reserve) — such a pass (`finalizeBodies`, `solveBullets`) refreshes its views unconditionally at its own top.

**Singleton, single-live-world.** The kernel is a singleton (one instance, one linear memory for every consumer). Resident state makes interleaved stepping of two live worlds cross-corrupt, so the wrapper tracks the owning world (`claimResident`/`releaseResident`) and stepping another live world throws; destroy releases. Sequential worlds (the fixture suite's shape) work; real multi-world stays unsupported (MT shipped on the one shared memory).

**The dynamic tree is a flat node pool** (`engine/tree.ts`): one ArrayBuffer, 12 four-byte slots per node mirroring C's 48-B `b3TreeNode` (aabb f32 ×6, category hi/lo, children/userData union under the LEAF guard, parent/next, height|flags), dual `Float32Array`/`Int32Array` views — same AoS-record family as the body columns. The three broadphase pools + the pairSet arrays are **kernel-resident** (`rust/tumble/src/broad.rs`; views managed by `engine/broadcolumns.ts`): TS keeps the mutation algorithms + growth policy, the kernel owns the hot read passes (query DFS + median-split rebuild, `tree.rs`/`pairwork.rs`) — so **b3DynamicTree logic exists twice; an upstream tree sync ports to both**, pinned by the tree gold vectors. Every broadphase entry point guards `store.refreshIfStale()` (O(1) when fresh), keyed on `broadGen()` + byteLength — the generation catches a relocation with no `memory.grow`. Singleton world-reuse rules for kernel-read structures: **first-use zeroing** (`ensureResident` — a fresh world over the grow-only region else inherits phantom pair membership) and **probe with the TS logical `set.capacity`, never the region cap** (a prior world's high-water). The broad region anchors at the **raw 4-aligned** manifold top — never align16 a region base over a variable-mod-16 anchor when the preserve-copy treats the base column as anchor-fixed (`kernel.test.ts` pins the orphaning sequence). Two traps the tree comments pin — both scoped to the *private-buffer* path (standalone/test trees + compound inner trees; resident pools preserve bytes via the kernel memmove): grow must copy through the **Int32 view** (a Float32 copy canonicalizes NaN-word category bits like `0xffffffff`), and never cache `nf`/`ni` across `allocateNode` (grow reallocates the buffer). Fat AABBs write through to the resident column inline at every write site — there is no lazy dirty/flush sync.

**Scratch-pool rules.** Copy-don't-alias shared refs into scratch; swap-remove via an index array. The awake body column is **dense-by-`localIndex`**, so every awake-set `swapRemove` (destroyBody, transferBody-out, trySleepIsland) copies the last record into the freed slot, mirroring the JS array compaction — a retained view is index-fixed, so fetch fresh by `localIndex`, never cache across a migration point (`trySleepIsland` migrates mid-solve).

**The convex/recycle partition.** The kernel *narrowphase* handles every convex contact — including static-involved: transforms/geometry arrive via dispatch records TS writes (`writeDispatchXf`/`writeDispatchGeom`), not via resident columns, so the static side needs no column. **The collide partition is incremental**: every awake contact lives in exactly one of `world.awakeRecycleContacts` (dynamic-dynamic direct-convex, both bodies awake → the batched kernel recycle pass) / `awakeOtherContacts` (static / mesh / compound / sleeping-partner → the TS per-contact walk), maintained at the event set — contact create/destroy (`contact.ts`), body wake (`wakeSolverSet`), sleep (`trySleepIsland`), transfer (`transferBody`) — never gathered per step. `recycleStable` is eligibility's per-contact-constant half, fixed at create; the only mutable input is body `setIndex`. Statics live in the static solver set; awake kinematics ARE resident and take the fast path. `bodySimIndexA/B` are **stored, not recomputed at read** (recompute-at-read doubled the random-access body-record traffic — a measured net regression until this flipped), propagated at every awake-set localIndex swap-remove via `writeBodySimIndex`; a **sleeping side is stale by design and unread until wake** — any new reader must be gated on wake. **Any new site that flips a body's `setIndex` or reassigns an awake body's `localIndex` must call `reclassifyBodyContacts`/`writeBodySimIndex`**; the first-principles partition invariant test (`step.test.ts` `assertPartition`) is the net. Static-involved contacts recycle via serial TS `tryRecycle` — measured 0.74% of large_pyramid contacts; a static snapshot column to move them was investigated and refuted 2026-07-12 (win inside noise; and note `body.setTransform` can teleport a static, so any such column would need that write site). Bit-exact-safe because recycle/narrowphase are per-contact independent, so partition and order don't affect results.

**Allocator trap.** No `Vec` / std allocation in the live wasm path — the std heap sits at `__heap_base` = the manual column arena, so an alloc corrupts columns and can `memory.grow` mid-call. Use fixed stack buffers (box3d's `LocalManifold` + `GEOM_CAPACITY` are 32, not 8). Cold a **destroyed** contact's directory cache on `freeSlot` (not a not-touching one — a recycled contactId else inherits a stale separating axis); `flush` zeroes the new cache region on grow (the singleton reuses memory across worlds).

**Why the pair query isn't parallel** (verdict 2026-07): fusion into an existing wake round is blocked by the serial filter+`createContact` gate (the recycle round reads the new contacts), so a parallel query costs a third wake round (~0.06–0.08 ms at t4) against ~0.18 ms compute at 565 moved/step — net negative, and worse on sleeping scenes below ~180–240 moved. Serial in-kernel is the design; don't re-propose without new evidence.

**Why no FMA / platform levers** (verified 2026-07-15): relaxed-SIMD FMA is nondeterministic by design and the C reference builds `-ffp-contract=off`, so the fixture contract forbids it regardless of engine support; memory64 loses guard-page bounds-check elision (10–100% slower); shared-everything-threads unshipped; `Atomics.pause`/branch hints marginal. No platform feature moves the gap — work placement and overhead removal only.

**Joint-port traps.** (1) **Write back solve-recomputed scratch the next substep reads.** A C solve that writes a prepared field back to the joint struct (`joint->perpAxisX/Y` in parallel + revolute) must write back in the kernel too — the next substep's warm-start reads it, and the isolated gold tier (single prepare→warm→solve) structurally cannot catch the omission; only a multi-substep fixture does. (2) **Explicitly zero conditionally-written prepare scratch when its feature is off** — the transient joint column is not zero-initialized (spherical's disabled-limit axes were the case). (3) A "diverges at t=N only" MT claim is almost always misdiagnosed: the kernel path is thread-count-independent by construction, so check whether OTHER thread counts also diverge vs serial before treating it as a framework bug — it's nearly always kernel-vs-serial parity. (4) A comment-only `.rs` edit that shifts line numbers legitimately changes both wasm artifacts — Rust bakes panic `file:line` into the data section (`strip` doesn't remove it); commit the rebuilt artifacts, it's not build nondeterminism.

## Multithreading

The hot path runs across a Web-Worker + `SharedArrayBuffer` pool over the wasm kernel. **MT is the default, not a level.** `init()` resolves the best the host affords; `init({ threads })` is the advanced escape (`0` forces single-thread, `n` overrides, clamped to the link bound); `threads()` introspects the resolved count (re-exported on `@dylanebert/shallot/tumble/core`). The auto count is **4** (`AUTO_THREADS` in `kernel.ts`) — flat, deliberately *not* scale-aware: more threads regress small scenes on wake cost, so "all cores" would ship a regression. Bit-exactness makes threads a pure perf dial with nothing to configure, which is why it's a property of the engine and not a knob a consumer picks (in shallot, tumble vs AVBD stays the only physics choice — threads never surface as engine API).

**Resolution table** (`resolve(threads, host)` in `kernel.ts`, a pure decision table so a unit test fakes the host): standalone (bun/node/deno — `SharedArrayBuffer` unconditional; deno routes here by construction but is *unverified* — run the exit-test consumer under deno before advertising it) → MT; a **cross-origin-isolated** browser → MT; a browser without isolation → single-thread after **one plain `console.log`** naming the missing COOP/COEP headers (host-actionable, loud, expected — the fix is the host's, not the caller's; per branding, the log points at the diagnostic, never the brand); explicit `0` → single-thread, no log. `host()` keys standalone on `process.versions.node`, the *same* predicate the pool uses to spawn — a UA-shaped check would misroute deno.

**Deployment.** A browser only multithreads when it can hold a shared `WebAssembly.Memory`, which needs cross-origin isolation: serve `Cross-Origin-Opener-Policy: same-origin` + `Cross-Origin-Embedder-Policy: require-corp`. Every shallot serve surface sends them (the `CROSS_ORIGIN_ISOLATION` constant in `src/project/vite.ts`, applied to `shallot dev`/preview/`verify` boots and both native webviews). A static host that can't set headers (GitHub Pages) gets the single-thread kernel + one log — a documented fallback, never an error. The real-browser positive assert is the gym `backend` scenario's isolation gate (`bun bench --scenario backend --param backend=tumble`): `crossOriginIsolated` **and** `threads() > 1` — assert the positive; a silent ST fallback still renders fine and would hide a broken MT boot.

**Dual artifact.** The single-thread kernel stays byte-identical on the *stable* toolchain — zero risk to the shipping default. The MT kernel is a **second base64 module** (`engine/kernel.shared.wasm.ts`) behind a dynamic `import()` in `multi()`, so an ST consumer never parses it. It needs shared memory + atomics (`+atomics,+bulk-memory,+mutable-globals`, `--import-memory --shared-memory`), which LLD only links together, and std rebuilt with atomics (nightly `-Z build-std`; the kernel touches no std API, so it's a toolchain switch, not a code change). One `bun run scripts/build-tumble-kernel.ts` emits both (separate `--target-dir`); the staged-solve code rides a cargo `--features` flag so the block table never bloats the ST artifact. **The full recipe + link args are codified in that script — don't restate them; edit there.**

**Bootstrap.** Each worker instantiates against the shared memory, then sets its own `__stack_pointer` to its 256 KiB slice and calls `__wasm_init_tls(tlsPtr)` (`__tls_size` is 8, not 0 — build-std links std's TLS regardless). **Main instantiates first and completes before any worker spawns** — a losing CAS in LLD's start function traps on `Atomics.wait` on a browser main thread, a browser-only trap bun/node can't catch, so the loader encodes the ordering. Worker stacks live in the *low* arena below the grow-only column arena (a stack above it would be relocated by a grow); `-zstack-size` (3 MiB) partitions main's 1 MiB + worker 256 KiB slices, so **`threads` is link-time bounded** (→ 7 workers). Workers are `unref`'d once at boot — that (not any `shutdown()`) is what lets a standalone script exit cleanly (`test:exit` gates it, node + bun).

**Concurrency invariants** (what keeps bit-exactness + memory safety):
- **No `memory.grow` / region relocation while workers are active.** Every `reserve*` runs pre-fork on the main thread with the arena already sized; parallel phases touch pre-reserved columns only. Each outer phase (`rust/tumble/src/parfor.rs`) has its own build+run pair forked *after* its own `reserve*`. Held by construction (build-before-wake, reserves pre-solve) AND asserted at runtime: `pool.run` snapshots the shared memory byteLength at wake and throws on mismatch at join (per-round, non-latching).
- **Shared memory never detaches.** `grow` hands back a *new* `SharedArrayBuffer` object aliasing the same backing store, so a worker's stale view still reads/writes the correct physical bytes and only misses the new tail. Views over the shared path key staleness on `memory.buffer.byteLength` (`sharedBytes()`), **never** on the `length === 0` detachment guard (which never fires there).
- **Park/wake.** Workers park in JS `Atomics.wait` between steps, woken once per pool round (2 rounds/step steady-state: recycle, then the staged solve with finalize fused as its terminal stage — `stages.rs`); stage transitions *inside* the solve are wasm-side stable-atomic spins (`core::sync::atomic` — load/store/cmpxchg/fetch_add only; the wait/notify intrinsics are unstable in Rust and would break the stable ST build). `pool.run` busy-waits synchronously between wake and join, so no `memory.grow` and no exit check can interleave mid-solve.
- **A worker fault poisons the kernel**, it doesn't just terminate survivors: the pool drives state that outlives the step (the staged solve's impulses, the persistent manifold store + contact directory), so stepping on single-threaded would run off a half-written store. The `dead` flag makes every later `kernel()` throw. A trap is a kernel bug (out-of-bounds column access), not a recoverable condition.
- **Thread-count independence is by construction, not hope**: within a color no two constraints share a body (blocks are write-disjoint), overflow + contact-creation stay serial in creation order, no reduction depends on worker identity. **Same hashes at any thread count is the contract** — the fixtures were generated on the serial path, so the existing gate applies to MT unchanged.

**One MT behavior the hash can't see:** joint events. Fixtures hash world state; events are behavioral post-step arrays, so the MT path flags joint events post-`readbackJointImpulses` (`flagJointEvent`) where the serial path flags per-substep. Divergence is possible only for a nonzero break threshold crossed transiently mid-substep. The standing coverage is the event cases in the process-tier fixture runner (`engine/step.fixture.ts`), asserted at ST, t2, and t8 — the kernel singleton's thread latch forces them into that harness, not `bun test`.

**Gates** (for any MT-touching change, from `packages/shallot`): `bun run test:fixture` (single-thread), `bun run test:fixture:mt` (shared artifact; `TUMBLE_THREADS` sets the count, default 4) at **t=2** (max barrier-interleave) and **t=8** (max block-claim scatter), `bun run test:fixture:auto` (the bare `init()` auto path), and `bun run test:exit` (no-`shutdown()` lifecycle: bundles a real engine consumer — the bare barrel tree-shakes to nothing — exit 0 on node + bun, threads > 1). The repo-root `bun run test:full` folds the single-thread fixture pass into the standard pre-commit gate. There is no in-repo C-comparison throughput bench (that died with the standalone package); perf reads ride the gym physics scenarios (`bun bench --scenario pile|backend`, `scripts/physics-bench.ts` at the repo root). Two perf disciplines survive it: decompose a step total into (serial TS tail, parallel kernel) before reading a pool comparison as a pool problem; and judge any perf claim by a same-window interleaved A/B against the pre-change commit — a green-gated, review-clean diff can still be a net perf regression, and any change whose win rests on moving/removing memory traffic gets the A/B before it lands.

## Fixtures

The 53 scene fixtures are **committed** at `tests/tumble/fixtures/` — per-step world-state hashes generated by the C reference, making the bit-exact contract self-contained: runnable anywhere, no C toolchain, no sibling checkout. (The runner reports 55 tests: the 53 scenes plus two joint-event behavior tests declared in the same fixture file.) Provenance (pin SHA, generator config) and the mint recipe live in that directory's `README.md` — the one source of truth; read it before touching fixtures. `scripts/run-tumble-fixtures.ts` replays them through `engine/step.fixture.ts`. Never hand-edit a fixture: it is C-generated truth, and a mismatch is an engine bug, not a fixture to adjust. Small gold vectors (`engine/*.gold.json`) are committed beside their tests.

Minting *new* fixtures or golds requires the box3d reference checkout — `reference/box3d` beside the shallot repo (the engine author's workspace layout), branch `harness`, plus cmake and a C toolchain. `bun run gen-tumble-fixtures` regenerates the scene fixtures; `bun run gen-tumble-gold <name>` regenerates one gold vector. Absent the reference, both error honestly — the expected state of a plain shallot checkout: an outside contributor cannot mint new fixtures, and doesn't need to, because every engine change validates against the committed set.

Scene conventions (for a new scene at a deliberate sync):

- **Benchmark scenes replay at debug scale** — same builders, same creation order, reduced counts (full sizes are infeasible as per-step gates). Full-scale throughput is the gym's job, not a fixture's.
- Per-step actions (mid-replay spawns, kinematic driving) use the `SceneStepFn` hook, called before `b3World_Step` exactly as the benchmark app does; extra scene data rides the `emitAux` JSON hook.
- Port-side scene builders **reuse the ported math helpers** (`quat.fromAxisAngle` etc.) and mirror the C expression tree left-associatively — constructed inputs must be bit-identical before the solver comparison means anything.
- Add a scene in the C generator (`fixtures/gen.c` + `g_scenes` in the reference checkout), rebuild via `gen-tumble-fixtures`.

## The sample-gold corpus and its standing gates

The second committed-truth corpus, beside the C fixtures: 40 gold trajectories at `tests/tumble/samples/` — per-sample initial snapshot + per-step `hashWorldState`, minted from the retired tumble.js sample corpus (provenance + frozen-truth status in that directory's README). Each gym twin scenario (`examples/gym/src/scenarios/tumble-*`) reproduces its sample's `build()`/`update()` through `tumble/core` and must replay its gold **bit-exact**; same never-hand-edit status as the fixtures — a mismatch is an engine or authoring bug, never a gold to adjust. The gym host layer (`examples/gym/src/tumble-*.ts`: oracle, grab, solids walk, overlay, watch instruments) derives all visuals from the world's own shape data — never hand-author a twin's visuals beside its physics.

Standing gates, all from the repo root:

- **`bun run test:gym`** — headless: the gold oracle (bit-exact replay per twin, one world per child process — the kernel singleton's grow-only regions make sequential rich worlds order-sensitive) + the host-layer unit tests. Rides `test:full`. Run after any engine, host-layer, or twin change.
- **`bun run bench:tumble`** — every twin loads and passes on real GPU over the bridge: one bridge session, page-per-scenario, per-scenario stall budget. Run after corpus-wide or render-path changes. **Never run two bridge consumers concurrently** — the bridge is one shared host browser; concurrent runs kill each other with connection-refused.
- **`bun run tumble:interaction`** — the dispatched-pointer tier on real GPU: synthetic `PointerEvent`s through the real input listeners; drag envelopes vs a pre-grab motion baseline, grab-no-launch, visual-draws + visual-solids (exact-count vs the world's own solid count). Run after grab / solids / overlay / input changes.
- **`bun run tumble:repro --gate`** — the browser-trusted-input tier (`page.mouse` over the bridge): violent-whip no-fling smoke, `--inject far` detector-intact check, and the standing invariants — non-finite scan on poses/transforms plus `staticBreach` (the build-time-registered static set asserted against live broadphase queries — an independent truth, deliberately not a derived-vs-drawn comparison, which is circular when both sides read the same source). Run beside `tumble:interaction` when a change touches anything between input and pixels.

Both interaction tiers exist on purpose: dispatched synthetic events are structurally unable to produce real input dynamics (browser-trusted input reproduced in one run a corruption dispatched probes never fired), so neither tier substitutes for the other.

## Upstream pin and sync

The engine ports a *specific* box3d commit, not box3d HEAD. box3d is a young pre-1.0 engine (one `v0.1.0` tag, actively renaming public API), so there is no upstream LTS to track — the frozen pin **is** the stable point. The pin lives in `engine/upstream.json` (provenance metadata); today it's `29bf523` (box3d `v0.1.0` + the Fixes 01–02 batches).

- **Default is freeze — permanently, by default.** The pin's whole value is the bit-exact contract: every committed fixture verified against that one SHA. Re-validating against a moving alpha is pure cost, so the pin moves only at a deliberate sync.
- **Sync only to a box3d coherence boundary** — a `Fixes 0N` batch commit or a tag, never a feature/draw commit or raw HEAD. box3d's real cadence is periodic `Fixes 0N` batches; landing on one keeps the pin at a point the maintainer treated as coherent.
- **Sync only when the boundary carries a fix the port can observe.** The engine ships no renderer, no build system (hand-written TS, not emscripten), no recording — so solver / contact / collision / mass / shape *math* can matter, plus (since MT ported box3d's task machinery) `scheduler.c`/`solver.c` staging changes. Docs, build flags, debug-draw, sample input, emscripten, recording changes are null for the port; don't spend a regen cycle on them. (Worked example: across `29bf523..540ea38`, four commits, the only engine-relevant delta was box3d #35 — explicit-mass bodies skipping CCD — deferred.)
- **Sync procedure** (when a boundary earns it — requires the reference checkout, see Fixtures): rebase the private `harness` fork onto the boundary SHA → regenerate the fixtures (`bun run gen-tumble-fixtures`) and re-verify bit-exact → port the engine delta op-for-op under the fixture gate → regenerate any affected gold vectors (`bun run gen-tumble-gold <name>`) → bump `engine/upstream.json` **and the pin line in `tests/tumble/fixtures/README.md`**.

## Test tiers

Follow the suffix convention:

- `*.test.ts` — fast unit tier, auto-discovered by `bun test`.
- `*.fixture.ts` — heavy bit-exact gates, run by `scripts/run-tumble-fixtures.ts` (`bun run test:fixture`), excluded from `bun test` by suffix.
- `*.lab.ts` — scratch investigation, never run in CI.

Tolerances are derived, never tuned — and the hash gate has none; it's equality. Any test file that boots the MT pool (`init()`) must `afterAll(shutdown)` — the kernel is a process-wide singleton, and a latched thread count / live pool leaks into sibling test files (per-file discipline, not an enforced invariant).

**Kernel-change checklist:** `cargo test` in `rust/tumble` green (native gold vectors) **and** `bun run scripts/build-tumble-kernel.ts` re-inlined the wasm — both `engine/kernel*.wasm.ts` are committed build artifacts (the build is deterministic, so a stale one shows as a byte diff). A pure-data-movement kernel change (e.g. the gather rewrite) needs no fixture regen; anything touching the arithmetic expression tree or solve order does.

Internals stay op-for-op faithful to the C where arithmetic matters; the API surface is idiomatic TS.
